**“最长公共前缀的长度”**——你眉头一皱——“这题名怎么听着像两个失散多年的兄弟在认亲？”

系统也不废话，直接甩出两个数组当例子：`arr1 = [1, 10, 100]`，`arr2 = [1000]`。末了还贴心地补了一段人话解释——“比如 `100` 和 `1000`。`100` 的前缀有 `1`、`10`、`100`，`1000` 的前缀有 `1`、`10`、`100`、`1000`。它们的公共前缀有 `1`、`10`、`100`，其中最长的是 `100`，长度是 3。“
”而你的任务，就是在两个数组里，找出属于 `arr1` 的整数 `x` 和属于 `arr2` 的整数 `y` 组成的所有数对`(x, y)` 之中最长的公共前缀的长度。如果根本没有公共前缀，就返回 0。”

你一拍脑袋：“这不就两层循环的事儿，把 `arr1` 每个数和 `arr2` 每个数都比一遍，找出最长的前缀不就行了！”

欲敲键盘之时，你忽然觉得事出简单必有坑——出题人放过谁？于是你瞄了一眼题目最下方，用最不起眼的字体印着的数据范围：

`1 <= arr1.length, arr2.length <= 5 * 10^4`  
`1 <= arr1[i], arr2[i] <= 10^8`

“靠，”你差点把鼠标扔出去，“这是要让全江湖的人互相滴血认亲啊！”

两个数组各五万个数，每个数还能大到一亿：暴力配对就是五万乘五万——二十五亿次比较——这要是提交上去，得到的不是 AC，而将是 TLE 的嘲讽。

你把脑子里那套双重循环扔进回收站，深吸一口气。昨天用频次数组和两个整数 O(n) 解决战斗，靠的就是四个字——**空间换时间**。今天这道题，暴力穷举是死胡同，必须换个思路。

------

### 第一幕：登记家谱+认亲查询

你盯着“前缀”两个字，脑子里闪过一个念头——一个数字的前缀，不就是它的“家族祖先”吗？

`12345` 往上追溯，祖先依次是：`1234`、`123`、`12`、`1`。每一位都比上一位短一截，但血脉相连——全是从最左边开始的亲祖宗。找两个数的“最长公共前缀”，本质上就是在两个数字的族谱里，找到那个辈分最近（数值最大/位数最多）的共同祖先啊。

“**预处理家谱 + 认亲查询**！”你一拍大腿：
“先把 `arr1` 里**每个数的每一个祖先（前缀）**——从一代目到最近的血亲——全部登记到一本**‘总家谱’（哈希集合）**里。然后让 `arr2` 的每个数来翻这本总家谱。对上了，就是公共祖先；所有对上的里面，谁辈分最近，谁就是最长公共前缀。”

这就好比：
`arr1` 先把家谱亮出来：“这是我的家谱，所有数字的所有祖先名字全在这本册子上。”
`arr2` 拿着自己的祖先名单过来：“那我挨个问，看你这本册子上能翻到我多远的祖宗。”

你快速心算了一笔账：**先建索引，再查索引**。建索引是 O(arr1 总前缀数)，查询是 O(arr2 总前缀数)，总复杂度 O((n+m) × L)，其中 L 是数字的最大位数——每个数最多 9 位（10^8 是九位数），一个数最多 9 个祖先。五万个数的 `arr1`，祖先总数最多 45 万。登记 45 万条家谱记录到哈希集合里，再查最多 45 万次认亲请求——这才几十万量级。跟二十五亿比起来，差了整整四个数量级，评测机应该连风扇不用转。

“能打！”你振臂一呼，准备登记家谱。

### 第二幕：古法誊抄——字符串切片的直白美学

第一版家谱，你决定走最直观的路线：**古法誊抄**。

把数字转成字符串，从左往右一刀一刀截，截一段就是一个祖先的名字，原原本本誊写到家谱上。属于“好记性不如烂笔头”的传统修谱派，语义直白，代码一看就懂。

```python
# 第一步：修家谱——arr1 所有数字的所有前缀登记入册
ancestors = set()
for num in arr1:
    s = str(num)                     # 数字转字符串，当祖先名册
    for i in range(1, len(s) + 1):
        ancestors.add(s[:i])         # 从第一个字符截到第 i 个，就是一位祖先
```

`arr1` 里每个数，转成字符串，然后从头一刀一刀切：`s[:1]`、`s[:2]`……一直到完整的 `s`。每一位祖先的大名都誊抄进总家谱 `ancestors`。至此，`arr1` 的全部数字血脉登记完毕——这个哈希集合就是你的家谱，O(1) 的查找速度，哈希函数在底层给你跑腿。

```python
# 第二部：认亲大会——遍历 arr2，贪心查家谱
max_len = 0 
for num in arr2:
    s = str(num)
    for i in range(len(s), 0, -1):    # 从辈分最近的祖先开始问，贪心！
        if s[:i] in ancestors:        # 哈希查找，O(1)，一击命中
            max_len = max(max_len, i)
            break                     # 合理剪枝——找到了就是当前数的最优解，后面的远祖不用看了
```

遍历 `arr2` 的每一个数，同样转字符串。关键一招：**从最长前缀开始查**——因为你要的是最长公共前缀，从近祖往远祖翻，一命中就是当前数字的最优解，后面的短前缀直接跳过。这是贪心的小心机——一步到位，不浪费感情，也不浪费 CPU 周期。

时间复杂度 O((n+m) × L)，空间复杂度 O(n × L)，L ≤ 9，最多 45 万条记录——评测机的风扇都呼不起来。

但你瞅着那些字符串切片，总觉得有些别扭： `s[:i]` 这行代码看似人畜无害，底层却在堆内存里新建字符串对象——有对象头，有内存分配，哈希的时候还得一个字符一个字符地喂给哈希函数算 Hash 值。虽然能过，但你骨子里的极客洁癖在隐隐作祟——

### 第三幕：心算法——整数除法的极致简约

“明明一个整数就能搞定的事，干嘛要让字符串这个中间商赚差价？”

你盯着“前缀”的本质，“数字本身不就是最好的前缀吗？为什么要转成字符串再切？直接用整数除法 // 10，从原数开始，每剥掉一次个位，就是一个新祖先。”

比如：`12345` 的祖先依次是：`12345`、`1234`、`123`、`12`、`1`——这个顺序天然从近祖排到远祖，简直是为贪心查询量身定做的——而整数，在 CPU 眼里又是一等公民：比较、除法、哈希，全是单条指令的活儿。字符串？那是软件层面的抽象，背后藏着一整套内存管理和字符编码的复杂语义。

这就好比：以前你是把祖先的名字誊抄在纸上（字符串），现在你直接把祖先的 DNA 样本（整数）存进寄存器。

```python
# 第一步：arr1 修家谱——所有数字的祖先整数入库
ancestors = set()
for num in arr1:
    while num > 0:
        ancestors.add(num)   # 当前完整数字就是一位祖先
        num //= 10           # 砍掉个位，追溯更远的祖先
```

登记进家谱的每一条记录都是一个原生整数——四个字节或八个字节，没有对象头，没有字符数组。除以 10 这种常数除法，编译器能直接优化成乘法+移位，整数的哈希是直接对数值取模——CPU都要爱上你的节奏。

```python
# 第二步：arr2 认亲——贪心从近祖往远祖查
max_prefix = 0
for num in arr2:
    while num:
        if num in ancestors:                   # 从近祖往远祖问
            max_prefix = max(max_prefix, num)  # 保留辈分最近的祖先
            break
        num //= 10                             # 砍掉个位，继续往上问
```

遍历 `arr2` 的每一个数，同样从原数开始，一刀一刀往上追溯。从完整数字一路追溯，祖先顺序天然降序，第一个命中的祖先就是当前数字的最长公共前缀。贪心被写进了循环的基因里。

最后一步，把 `max_prefix` 转换成长度：

```
return len(str(max_prefix)) if max_prefix else 0
```

唯一一次字符串转换，只是为了数一下找到的最长公共前缀有几位。如果 `max_prefix` 一直是 0，说明两家谱上没有一个记录对得上，返回 0。

你忍不住感叹：“两趟遍历，一个集合，一个 while，一把除十。这他妈才是写给机器的情书——每一行代码都直接映射到 CPU 指令，每一刀除十都精确命中一个祖先。”
