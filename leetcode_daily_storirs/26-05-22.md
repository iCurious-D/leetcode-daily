### 第一幕：分步剑法——稳扎稳打的老实人

你深吸一口气，决定先不想什么花里胡哨的：

> **既然数组被旋转了，那就先找到旋转点（最小值），把数组重新切成两段有序的升序段，然后判断 target 落在哪一段，在那一半里规规矩矩做二分查找。**

这就像拿到一本被人从中间撕开、前后粘错的词典。要查一个词，最笨但最稳的办法就是——先翻到粘错的那个断口，搞清楚前半本和后半本各自是什么范围，然后再决定去哪一半里翻。

逻辑闭环，没有魔法，全靠基本功。你甚至觉得自己不是在解题，是在跑一个**二阶段流水线**。

**第一步：找旋转点——当初在酒馆里学的那招“红蓝染色找最小值”**

```python
def findMin(self, nums: List[int]) -> int:
    left, right = -1, len(nums) - 1
    while left + 1 < right:
        mid = (left + right) // 2
        if nums[mid] < nums[-1]:
            right = mid      # 蓝染右收：mid 比最后一个元素小，掉进了右半段（小数区）
        else:
            left = mid       # 赤染左推：mid 比最后一个元素大，还在左半段（大数区）
    return right
```

红蓝交界处，就是旋转点的位置——最小值的下标。这招你已经练到**肌肉记忆级复现**，闭着眼都能写。

**第二步：在有序段里找 target——“开区间二分查找标准模板”**

旋转点找到了，接下来就是在确定的升序段里老老实实找 target。

```python
def lower_bound(self, nums: List[int], left: int, right: int, target: int):
    while left + 1 < right:
        mid = (left + right) // 2
        if nums[mid] >= target:
            right = mid      # mid 的值大于等于 target，收缩右边界
        else:
            left = mid       # mid 的值小于 target，收缩左边界
    return right if nums[right] == target else -1
```

时间复杂度 O(log n)，稳如老狗，属于那种“能跑就行，别问我原理”的祖传代码。

**第三步：拼装上线，流水线作业**

你把两套招式往桌上一摊，开始流水线组装：

```python
def search(self, nums: List[int], target: int) -> int:
    min_id = self.findMin(nums)        # 第一步：定位旋转点
    if target > nums[-1]:
        # target 比最后一个元素大，说明 target 只能在左半段（较大段）
        # 直接去左半段 [0, min_id) 里搜
        return self.lower_bound(nums, -1, min_id, target)
    # 否则 target 一定在右半段 [min_id, n) 里
    return self.lower_bound(nums, min_id - 1, len(nums), target)
```

逻辑清晰，步骤分明，像极了酒馆老板切牛肉——先称重，再下刀，最后装盘。两个 O(log n) 串在一起还是 O(log n)。你觉得自己可以收工了。两步走怎么了？代码可读性好，团队协作友好，面试官看了都说稳。

但就在你洋洋得意、准备提交代码跑路的时候，背后传来一声冷笑。

### 第二幕：一体化剑法——读心术的极致

仙人从袖中抽出那把双刃剑，剑身上刻着四行神秘的咒语：

```python
def search(self, nums: List[int], target: int) -> int:
    left, right = -1, len(nums) - 1
    while left + 1 < right:
        mid = (left + right) // 2
        if target > nums[-1] >= nums[mid]:      # 情况1
            right = mid
        elif target <= nums[-1] < nums[mid]:    # 情况2
            left = mid
        elif nums[mid] >= target:               # 情况3
            right = mid
        else:                                   # 情况4
            left = mid
    return right if nums[right] == target else -1
```

你看得眼花缭乱，脑细胞直接栈溢出：“这四行 if-elif 是什么鬼？怎么还有链式比较？`target > nums[-1] >= nums[mid]` 这种写法在 Python 里虽然合法，但读起来比正则表达式还像天书！你这是把条件判断写成了一行流，是想挑战人类阅读理解的上限吗？"

仙人哈哈大笑，把剑往桌上一拍：“这就是**一体化剑法**的精髓——不找旋转点，不切段，在一次二分查找中，同时完成‘判断区间’和‘收缩边界’两个动作。你刚才的分步剑法，是先查户口再抓人；而一体化剑法，是**直接读心**。”

**🔸四条咒语的真相**

“拆解之前，先记住一个核心思想。”仙人竖起两根手指。

“旋转数组本质上就是**两个升序段拼在一起：\[ 较大段(红区) ][ 较小段(蓝区) ]**——而有一个**锚点**永远钉在那儿：**nums[-1]（最后一个元素）**。它是整个蓝区（较小段）的老大哥，也是红蓝分界的参照物。盯住它，就像盯住北极星，永远不会迷路。”

“ target 只有两种出身——要么比 nums[-1] 大（落入红区），要么比 nums[-1] 小或等于（落入蓝区）。每一次循环，你都同时回答两个问题：**mid 在哪儿？target 相对于 mid 在哪儿？**——答对了，就能安全砍掉一半搜索空间。”

仙人蘸酒，在桌上写下四条咒语的本质分类：

1. **跨区**：target 和 mid 一个在红一个在蓝 → 直接用锚点判断方向，一步到位。
2. **同区**：target 和 mid 掉进同一段升序数组 → 退化成普通二分查找。

“前两条负责拦截跨区这种复杂场景，后两条处理同区这种常规场景——**特殊情况提前处理，剩下的全是主流程**——这就是一体化的优雅。”

**🔸情况1：target > nums[-1] >= nums[mid]**

跨区：target 是红区高人，mid 是蓝区小弟（或刚好站在边界上）。
此时，target 一定在 mid 的**左边**（因为红区在数组前半部分，蓝区在后半部分），`right = mid`，右边界左移。

🔸**情况2-：target <= nums[-1] < nums[mid]**

跨区：target 是蓝区小弟（或恰好在边界），mid 是红区高人。
此时，target 一定在 mid 的**右边**（因为蓝区在数组后半部分）， `left = mid`，左边界右移。

“前两条负责跨区作战——target 和 mid 各站一个阵营。但如果前两个条件都不满足，那就说明target 和 mid掉进同一段了，就可以直接用标准的二分查找逻辑——”

**🔸情况3：nums[mid] >= target**

同区：target 和 mid 在同一段升序数组里，mid 的值已经大于等于 target 了，说明 target 在 mid 左边（或就是 mid 自己）， `right = mid`，**经典蓝染右收。**

**🔸情况4：else（即 nums[mid] < target）**

同区，mid 比 target 小，target 只能在 mid 的右边， `left = mid`，**经典赤染左推。**

你若有所思：“我懂了！情况1和2处理跨区，情况3和4处理同区。但我有一个问题——**这四条分支的优先级为什么这样排？** 我把后两条挪到前面行不行？”

仙人眼睛一亮：“好问题，问到点子上了。”

“你看前两条——`target > nums[-1] >= nums[mid]` 和 `target <= nums[-1] < nums[mid]`——它们**把 nums[-1] 嵌在了链式比较的正中间**。如果 target 和 mid 恰好在同一个半区，nums[-1] 就会起到天然的阻断作用，让前两个条件无法同时成立，于是自动 fall through 到第三、第四条。这就是优先级的设计逻辑——”

- **前两条处理跨区**：必须用 nums[-1] 做裁判。不满足跨区条件？丝滑进入同区逻辑。
- **后两条处理同区**：退化为普通二分，不需要 nums[-1] 参与。

“优先级绝对不能换。如果先写同区的条件，跨区时会出大乱子。”

”举个例子：target 在蓝区（小），mid 在红区（大），`nums[mid] >= target` 这个条件就是成立的——如果先执行第三条，就会错误地执行 `right = mid`，把左边正确的搜索空间一刀砍没。这叫什么？**假阳性误判**，程序能跑，但结果是错的，属于那种‘测一万条用例炸一条’的究极 bug。”

你倒吸一口凉气：“原来链式比较里的 `nums[-1]` 不仅是参照物，还是**天然的守卫条件**——它保证前两条只在跨区时触发，同区时自动让路。这设计太精妙了，简直像操作系统里的中断向量表，优先级排队，缺一不可。”

“正是。”仙人把双刃剑收回袖中，“这就是一体化剑法的精妙之处：**用条件表达式的顺序，隐式地处理了优先级逻辑，而不是靠额外的 if-else 嵌套。** 代码极简，但读起来需要动脑子——属于那种‘写得一时爽，review 火葬场’风格。”

### 终幕：一体化 vs 分步

你还是不甘心，追问道：“这样做真的比分步法快吗？我不信。两个 O(log n) 和一个 O(log n)，有区别？”

仙人点头，眼中闪过一丝狡黠："从渐进复杂度看，两者都是 O(log n)，这点你没说错。但一体化有两个优势：

1. **常数因子更小**：分步法需要两次完整的二分查找——先找旋转点，再找 target，两套 while 循环从头跑到尾。而一体化只需要一轮循环，虽然循环体内多了几个 if 判断，但总体迭代次数几乎减半。
2. **代码更紧凑**：不需要额外写 `findMin` 和 `lower_bound` 两个辅助函数，所有逻辑集中在一个函数里，减少了函数调用开销——毕竟 Python 的函数调用开销大得离谱，每多一层调用栈就多一层罪过。

“当然，代价是**可读性断崖式下跌**。这四条链式比较放在一起，对于一个凌晨三点还要 debug 的人来说，杀伤力堪比生产环境的核心代码没有注释——你看得懂每一行，但就是不知道它们在干什么。”

你追问："那在面试中，应该用哪种？"

仙人想了想，说："如果你能在 5 分钟内写完一体化且不出错，那当然用一体化。但如果你担心写错，**分步法是更安全的选择**——它逻辑清晰，每一步都能单独验证，即使写得慢一点，也能保证正确性，**AC 比炫技更重要。**"
