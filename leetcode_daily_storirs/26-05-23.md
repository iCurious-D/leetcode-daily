#### 第一幕：断崖计数器

你决定用最朴素的逻辑：**如果一个数组是有序数组轮转得到的，那么它最多只能有一个'断崖'（即下降的位置）。**

比方说：

- `[1, 2, 3, 4, 5]`：从头走到尾，0个断崖，全程非递减，规矩得像公务员排队。
- `[3, 4, 5, 1, 2]`： `5 → 1`  一个断崖，合法的旋转数组。
- `[2, 1, 3, 4]`：`2 → 1` 是一个断崖，`1 → 3` 升回来，`3 → 4` 也升——光看内部相邻对好像只有一个断崖？
  **但别忘了，数组是轮转的，首尾也算相邻**。`4 → 2` 又是一个断崖，2 个断崖 ，这数组根本不是正经旋转，而是乱序打散后的纯假货。

**合法旋转数组 = 至多一个断崖（`nums[i] > nums[i+1]` 出现不超过一次）。断崖超过一处，直接判假。**

你麻溜地掏出键盘，祭出计数器大法：

```python
def check(self, nums: List[int]) -> bool:
    n = len(nums)
    count = 0  # 断崖计数器
    for i in range(n):
        # 检查当前元素和下一个元素的相邻关系（注意：末尾的下一个是开头，用取模实现）
        if nums[i] > nums[(i + 1) % n]:
            count += 1
        # 断崖超过一处，直接判假
        if count > 1:
            return False
    return True
```

从头转到尾，**末尾用 `% n` 绕回头检查首尾衔接**。每遇一个坑 `nums[i] > nums[i+1]`，`count` 加一；`count` 一旦飙到 2，立刻 `return False`，看都懒得多看一眼。遍历完还没超标，断崖数是 0 或 1，就是合法旋转数组，放行通关。

你看着自己写的代码，心里涌起一种奇怪的成就感——这大概是你学算法以来写过的**最像人话的 O(n)**。

没有二分，没有左右指针，没有开区间闭区间的哲学纠结，只有一个 for 循环和一个计数器，朴实得像用算盘算 1+1。时间复杂度 O(n)，空间复杂度 O(1)，连个多余的数组都没开——比你的钱包还干净。

你甚至觉得这代码可以贴到菜市场当质检标语：**“一圈下来，坑不超过一个，就是好货。”**

#### 第二幕：is_sorted 状态机

“你的解法没错，” 熟悉的声音缥缈而至。

果然，还是那个神出鬼没的大仙：“断崖计数，逻辑清晰，跑起来也没毛病。但你不觉得——用一个 int 当计数器，太浪费了吗？”

你一愣：“浪费？一个 int 才四个字节，还能省？”

仙人摇摇头：“不是内存的问题，是**境界**的问题。你用计数器，是在说‘我要记住遇到了几个断崖’。但你再想想——这道题真的需要知道断崖的**个数**吗？”

仙人蘸酒，在桌上重写逻辑：

“只需要知道三件事：

1. **首尾之间有没有断崖**——这是初始状态。
2. **遍历内部相邻对时，有没有遇到断崖**——这是状态转移。
3. **如果已经遇到过一个断崖，再遇到第二个就直接判假**——这是终止条件。”

“这三件事，用**一个布尔变量**就够了。”

```python
def check(self, nums: List[int]) -> bool:
    is_sorted = nums[0] >= nums[-1]
    for x, y in pairwise(nums):
        if x > y:
            if not is_sorted:
                return False
            is_sorted = False
    return True
```

你盯着这段代码，脑内 AST 解析器直接栈溢出：“一个 `is_sorted` 布尔变量？`nums[0] >= nums[-1]` 的初始化又是什么鬼？”

仙人笑了：“这不是计数器，这是**状态机**。`is_sorted` 不是你想象中的‘数组是否排好序’，而是一个**断崖配额**状态——它代表‘你的断崖配额还有没有了’。”

**第一步：初始化配额**

```python
is_sorted = nums[0] >= nums[-1]
```

“这一行是整个状态机的起点。它在问：**首尾相接处，消耗了你的断崖配额吗？**”

- 如果 `nums[0] >= nums[-1]`：首元素大于等于尾元素，`尾→首`衔接处没有下降，也就是没有消耗你的断崖配额——轮转后的真排序数组正是如此：较大段被顶到前面，较小段甩到后面——` is_sorted = True`表示断崖配额还在，内部还能容纳1个断崖。
- 如果`nums[0] < nums[-1]`：首元素小于尾元素，`尾→首`相接处是下降，消耗了你的断崖配额————这只能发生在真数组根本没旋转或旋转了整圈回到原位（全局升序），也就是说，数组内部不允许出现任何下降—— `is_sorted = False`表示断崖配额已消耗，数组内部不允许出现任何下降，有下降直接判假。

**第二步：遍历相邻对，管理配额**

```python
for x, y in pairwise(nums):
    if x > y:
        if not is_sorted:
            return False
        is_sorted = False
```

“`pairwise(nums)` 把数组变成相邻对迭代器：`(nums[0], nums[1]), (nums[1], nums[2]), ..., (nums[n-2], nums[n-1])`。注意——它**不包含首尾衔接对**——这正是我们想要的，因为首尾已经在初始化时处理过了。接下来挨对检查即可：”

- 如果 `x <= y`（升序或持平），一切正常，状态机空转。
- 如果 `x > y`（遇到一个断崖），进入断崖处理逻辑：
  - **先查配额**：`if not is_sorted`——配额已用完，这是**第二个**断崖，直接 `return False`。
  - **配额还在**（`is_sorted == True`），说明这是遇到的第一个（也是唯一合法的一个）内部断崖，消耗配额——`is_sorted = False`，然后继续遍历。

“遍历结束还没触发 `return False`？说明内部至多一个断崖，且首尾配合得当，返回 `True`。”

你猛拍大腿：“妙啊！这不就是用一个布尔值模拟了‘断崖计数器’？`is_sorted` 从 `True` 翻成 `False`，就是计数器从 0 变 1；`if not is_sorted` 就是问计数器是不是已经到 1 了。**用一个 bool 的翻转替代了 int 的自增和比较，少一次加法，少一次比较。**

仙人点头：“没错。而且，`pairwise` 让你不需要像计数器版本那样手动 `% n` 绕回。**首尾衔接的检查被优雅地分离到初始化语句里**，遍历只关心内部相邻对。职责分离，代码更干净。”

他顿了顿，嘴角扬起一丝神秘的笑容：
“还有，`is_sorted` 这个名字你不觉得很有禅意吗——读起来像是‘数组是否有序’，实际上是‘你是否还有断崖配额’。不懂的人一看：哦，真排序就返回 true。懂的人一看：操，这变量是个状态机的核心。**一个变量名能同时骗过菜鸟和取悦高手，简直是命名艺术的巅峰。**”
