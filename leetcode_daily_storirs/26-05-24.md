#### 第一幕：两件法器

说是神器，其实就两样朴实无华的东西：一张备忘录符纸，一把递归镰刀。

**一：备忘录 `memo`（记忆数组）**

```python
memo = [-1] * n
```

`memo[i]` 代表从下标 `i` 起跳，一路顺着下坡滚到底，最多能踩中多少个台子（算上脚底下这个）。

初始化全 `-1` 当作“尚未计算”的哨兵——因为最少也能踩 1 个，用 `-1` 区分未访问，干净不出歧义。

**二：DFS 递归镰刀**

```python
def dfs(i):
    if memo[i] != -1:
        return memo[i]    # 算过了，直接交作业
    
    max_steps = 1   # 哪怕绝路一条，脚下这块地也算业绩
    # 向右探测，射程最远 d 步
    for x in range(1, d+1):
        j = i + x
        if j >= n:        # 撞墙，这里不可能有路了
            break
        if arr[j] >= arr[i]:  # 遇到高的，视线直接挡死
            break             # 连 j 都不能跳，更远的也甭想，直接 break
        max_steps = max(max_steps, 1 + dfs(j))            
    # 向左探测，镜像操作
    for x in range(1, d+1):
        j = i - x
        if j < 0:
            break
        if arr[j] >= arr[i]:
            break
        max_steps = max(max_steps, 1 + dfs(j))
        
    memo[i] = max_steps
    return max_steps
```

你一眼看穿那个“中间所有柱子必须更矮”的约束，根本就是系统白送你的剪枝大礼包——`arr[j] >= arr[i]` 不仅宣告当前 `j` 不可达，连更远的下标也被这个“视线遮挡”一次性作废，直接 `break`，后面的 `x` 都不用遍历，堪比你写二分查找时 `mid` 直接命中目标的爽感。

每个位置 `i` 左右各扫一遍射程，剩下的脏活累活全部甩给 `dfs(j)` 递归到底。每一层返回自己能铺出的最长下坡，上层再取个最大值 `+1`，标准得不能再标准的 DAG 上 DP模板。

探路逻辑写完，剩下就是挑山头了：

```python
# 枚举所有起点
ans = 0
for i in range(n):
    ans = max(ans, dfs(i)) 
    
return ans
```

所有下标都当一次下坡起点，打完收工。因为记忆化保底，每个状态只算一次，时间复杂度 O(n*d)，空间换时间，常数小得感人。系统想拿超时制裁你？它连 TLE 的 `T` 都还没渲染完你就已经 AC 了。

#### 第二幕：@cache 装饰器—— Pythonic 的魔法

老道长袖袍一挥，一段散发着python 之禅微光的代码在空中浮现 ：

```python
from functools import cache

class Solution:
    def maxJumps(self, arr: List[int], d: int) -> int:
        n = len(arr)
        
        @cache   # 自动记忆化，等价于你的整个 memo 体系
        def dfs(i: int) -> int:
            res = 1
            
            # 往左跳
            for j in range(i - 1, max(i - d - 1, -1), -1):
                if arr[j] >= arr[i]:
                    break
                res = max(res, dfs(j) + 1)
            
            # 往右跳
            for j in range(i + 1, min(i + d + 1, n)):
                if arr[j] >= arr[i]:
                    break
                res = max(res, dfs(j) + 1)
            
            return res
        
        # 枚举所有起点，一行 max 生成器搞定
        return max(dfs(i) for i in range(n))
```

老道指尖轻点 `@cache`：“这是 Python 3.9 起内置的装饰器，本质就是个 **无限容量的LRU 缓存**。你手写 `memo` 数组，说到底就是用一个列表模拟哈希映射，还得手动管理哨兵值——要是答案真可能是 `-1`，你就得换魔法数字，蠢得冒泡。而 `@cache` 直接交给解释器，内部用字典维护，有就有，没有就算，自动区分‘未计算’和‘计算结果’，干净得像雪后的空栈。”

他顿了顿，又说：“再看那俩 `for` 循环，你用 `range(1, d+1)` 再手动算 `j = i + x`，还在里面 `if j >= n: break`，一股 C 味。这里直接用 `range(i+1, min(i+d+1, n))`，把边界保护写进迭代器构造里，循环体只处理阻挡逻辑——**让语言帮你干脏活**，这才对得起 Guido van Rossum 祖师爷。”

你心底暗暗佩服，可还是嘴硬：“但数组随机访问是 O(1) 且常数极小，字典查找虽然均摊 O(1)，哈希计算毕竟有开销，这不是用性能换优雅吗？”

老道长仰天大笑：“好问题！`@cache` 底层字典确实比列表多了一点常数开销，但 Python 的字典优化早已出神入化，这点差异在绝大多数用例里根本测不出来——况且你代码里 `dfs` 调用次数也就 O(n*d)，远没到能榨干 CPU 的地步。真的遇上变态 case，你还可以用 `@lru_cache(maxsize=None)` 多些控制。退一步讲，就算真多了几毫秒，比起你维护 `memo` 时不小心写出边界 bug、调了半小时才发现 `-1` 污染数据的痛苦，这点开销连利息都算不上。**代码越短，bug 藏身的地方越少**——这是软件工程的硬通货，比 O(1) 常数还硬。”

你恍然大悟：“原来如此！手写 `memo` 时，我的确担心全局变量污染、递归栈帧里忘记引用外部数组之类的破事。`@cache` 直接绑在函数上，封装得死死的，连命名空间都清爽了。”

老道满意点头：“正是。而且这种写法还有一个隐藏福利——记忆化完全托管给装饰器后，你的 `dfs` 变成了一个**纯函数**，只依赖参数 `i` 和外部只读常量，无副作用，可重复入，线程安全。这种纯函数思维，是你以后写函数式编程、玩并行计算时千金不换的童子功。“
