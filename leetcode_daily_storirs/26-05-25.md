> 给你二进制字符串 `s` 和两个整数 `minJump` 和 `maxJump`。  
> 一开始你在下标 `0`，且该位置的值一定为 `'0'`。  
> 你可以从 `i` 跳到 `j`，当且仅当：  
> `i + minJump <= j <= min(i + maxJump, s.length - 1)`  
> **且** `s[j] == '0'`。  
> 目标：判断你能否到达终点 `s.length - 1`，能则返回 `true`，否则 `false`。

“……每次跳跃有一个区间 [i + minJump, i + maxJump]......” 你冷笑一声，“这不就是**带跳跃范围的连通性检测**嘛！跳跃游戏 III 是裸 BFS，IV 是带传送门的 BFS，V 是 DAG 最长路。到了 VII，本质还是可达性，祖传 BFS 再次出征，稳如老狗。”

```python
from collections import deque

def canReach(s: str, minJump: int, maxJump: int) -> bool:
    n = len(s)
    if s[-1] == '1':     # 终点是岩浆，开局即死局
        return False

    q = deque([0])
    seen = {0}

    while q:
        i = q.popleft()
        if i == n - 1:
            return True
        
        # 枚举所有可能的落点
        start = i + minJump
        end = min(i + maxJump, n - 1)
        for j in range(start, end + 1):
            if s[j] == '0' and j not in seen:
                seen.add(j)
                q.append(j)
    
    return False
```

### 第一套心法：滑动窗口BFS 

“你仔细看，”道长指着石板上的题目，指尖在“`i + minJump <= j`”那行字上重重一点，“这题有一个极其好用的性质，大到可以当外挂用——你只能**往前跳**，索引严格递增。”

“索引单调递增意味着什么？意味着 BFS 队列里的节点，它们的下标天然就是升序排列的。先入队的下标一定小，后入队的一定大。已经被访问过的 `j`，对于队列里后面那些 `i` 来说，永远是一去不回的‘前辈’，绝不可能再被需要。所以你缺的不是剪枝，是一个**全局游标 far**，用来记录已经扫描过的最大下标——也就是你的 BFS 推土机铲斗推到哪儿了。”

他顿了顿，眼中闪过一道 O(n) 的寒芒：

“每次处理节点 `i` 时，你不需要从 `i + minJump` 开始老老实实地扫，而是直接从 `max(i + minJump, far + 1)` 起步——`far + 1` 意味着‘上次铲到的位置之后第一个还没看过的地方’。一路扫到 `min(i + maxJump, n-1)`，把路上碰到的所有 `'0'` 统统塞进队列，然后更新 `far`。这样一来，**整个数组里每个下标至多被扫描一次**，总扫描次数严格 O(n)，这就叫**均摊 O(1) 的邻域展开**。稠密图？不存在的，你的扫描区间根本就不重叠。”

你一拍大腿：“卧槽，这不就是滑动窗口嘛！BFS 是推土机，`far` 是铲斗的前缘，铲过的地绝不回头铲第二遍！每个格子进一次 `for` 循环就判生死，拒绝二次消费！”

在前辈的点化下，你键盘冒火星子，飞速敲出优化版：

```python
from collections import deque

def canReach(s: str, minJump: int, maxJump: int) -> bool:
    n = len(s)
    if s[-1] == '1':
        return False

    q = deque([0])
    far = 0   # 全局最远已扫描下标，单调不减

    while q:
        i = q.popleft()
        if i == n - 1:
            return True
        
        # 只扫描还没有被 far 覆盖的区间
        left = max(i + minJump, far + 1)
        right = min(i + maxJump, n - 1)
        
        for j in range(left, right + 1):
            if s[j] == '0':
                q.append(j)
                if j == n - 1:   # 提前收工
                    return True
        # 更新铲斗位置 far，无论这个区间里有没有 '0'，这些下标都已经处理过了
        far = max(far, right)

    return False
```

你盯着这个 `far`，感叹道：“这玩意儿本质就是个**单调全局游标**，把 BFS 每个节点各自为政的邻域扫描，合并成了一次完整的线性遍历。每个格子至多被 visit 一次，总复杂度 O(n)，空间 O(n)。这就是用指针把稠密图活活削成稀疏图，降维打击啊！”

道长点头：“孺子可教。但你可知，这题还有另一种更‘声明式’的写法——你甚至不需要队列。”

### 第二套心法：前缀和优化DP 

“你的 BFS 是正向思考——‘从 i 能跳到哪些 j？’，从而导致你要遍历很多条边。现在，我们换个角度：**‘哪些 j 能跳到 i？’** 把 ‘推’ 变成 ‘拉’ 。”

道长伸手在空气中画了一张表：“不用队列，不用 `seen` 集合，直接定义 `dp[i]` 表示下标 `i` 是否可达。“

” 转移方程一句话：`dp[i] = (s[i] == '0') AND (区间 [i - maxJump, i - minJump] 内存在至少一个可达点)`——这本质上是一个**区间存在性查询**——你不是在问‘从 `i` 出发能到哪儿’，而是在问 ‘有没有人能到 `i`’。从前驱的视角看，**每个 i 的可达性只取决于前面一个固定长度的窗口**。”

你抢答道：“可是暴力扫描这个区间去检查是否有 `True`，每个 i 又得扫 O(d) 个元素，复杂度还是 O(n*d) 啊！”

道长微微一笑：“不错，这正是问题所在。但‘区间内是否存在 `True`’这个问题，可以用**前缀和数组** O(1) 回答——只要你预先维护 `dp` 的前缀和 `pre`，`pre[k]` 表示 `dp[0..k-1]` 中 `True` 的个数。那么区间 `[i-maxJump, i-minJump]` 内 `True` 的个数，就是 `pre[i-minJump+1] - pre[max(i-maxJump, 0)]`。只要这个差值大于 0，说明前面有落脚点能跳到当前位置。这就叫**前缀和优化 DP**——把区间遍历的 O(d) 压成 O(1)，直接从 O(n*d) 砍到 O(n)。”

老道长袖袍一挥，一段散发着数学光辉的代码在空中徐徐展开：

```python
def canReach(s: str, minJump: int, maxJump: int) -> bool:
    n = len(s)
    if s[-1] == '1':
        return False

    dp = [False] * n
    dp[0] = True
    # pre[i] 表示 dp[0..i-1] 中 True 的个数
    pre = [0] * (n + 1)
    pre[1] = 1   # dp[0] 是 True

    for i in range(1, n):
        # 1. 区间 [i-maxJump, i-minJump] 内是否有可达点
        dp[i] = (i>=minJump and s[i] == '0' and
                 pre[i - minJump + 1] > pre[max(i - maxJump, 0)])   
        # 2. 更新前缀和
        pre[i+1] = pre[i] + dp[i]

    return dp[-1]
```

道长轻抚长须：“你看这行关键判定——`pre[i - minJump + 1] > pre[max(i - maxJump, 0)]`，它等价于区间内 `True` 的个数大于 0，直接省去了显式做减法，常数更小，可读性嘛——懂的人都懂。每个位置 O(1) 转移，全程 O(n) 时间，O(n) 空间，干净得像一道晨光。”

你盯着那段代码，脑子里豁然开朗：“原来如此！前缀和把区间遍历的 O(d) 降到了 O(1)，总复杂度从 O(n*d) 直接一刀砍成 O(n)！这不是优化，这是算法的降维打击！前缀和 + DP 简直是区间可达性问题的银弹，一发入魂。”

老道长点头：“正是。而且这种写法还有一个隐藏优势——你不需要维护队列，也不需要操心 `far` 指针的边界条件，更不用担心 `seen` 集合的内存占用。所有状态都通过 `dp` 数组和前缀和线性推进，**声明式编程**——你只管定义‘什么是可达’，不用教计算机‘怎么去找可达点’。数据流清晰得像一条笔直的流水线，每一步都像纯函数，给定输入就铁定输出，不抛异常、没有副作用、单元测试闭着眼睛都能写。”

你双手合十，心悦诚服：“感谢前辈！滑动窗口 BFS，是命令式、显式队列的推土机打法，铲过的地绝不回头，叫**滑窗流**；前缀和 DP，是声明式、反向查询的优雅打法，区间存在性 O(1) 判定，叫**前缀流**。双法得兼，区间优化轻松拿捏。”
