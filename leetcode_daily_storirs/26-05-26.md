## 🎮 菜鸟勇者的字母大陆历险记1

我叫小白，一个连技能树都没点全的萌新勇者。今天村长把我喊过去，说字母大陆出了一批“双面间谍”——有些字母既用小写形态潜伏，又用大写形态搞事情。我的任务很简单：统计出到底有多少个字母同时拥有大小写两种形态。

村长把任务卷轴塞给我的时候特意强调：“单词长度不超过50，你要用最少的资源完成任务。记住，你是个勇者，不是氪金战士，每一行代码的复杂度都得给我省着点用。”

我拍拍胸脯：“不就是数大小写配对嘛，我分分钟搞定！”

#### 第一战：大力出奇迹的暴力流

我来到字母大陆入口，前面横着一支杂乱的字符串军队：“AbCdEfGa”。怪物们飘浮在空中，有的挺着大写肚子，有的缩成小写虾米。我二话不说，掏出《Python 魔法书》——这还不简单？先把整支队伍丢进集合去个重，然后把26个小写字母全审一遍，哪个字母自己和它的大写兄弟都在集合里，就给它记上一笔。

咒语随手捻来：

```python
s = set(word)
count = 0
for c in "abcdefghijklmnopqrstuvwxyz":
    if c in s and c.upper() in s:
        count += 1
return count
```

“O(n)时间建集合，再花常数52次哈希查找，一共也就五十来次操作，稳得很！”我叉着腰，觉得自己是个天才。

结果一个老勇者叼着烟斗飘过来，瞄了一眼就开始喷：“你这叫什么？每查一个字母都要在集合里翻两遍！这就像去食堂打饭，明明扫一眼就知道今天有啥菜，还非要每个窗口都问两遍 ‘阿姨红烧肉有吗？阿姨糖醋排骨有吗？’ 阿姨没拿勺子敲你都是仁慈。”

我一愣：“可我这是常数级啊……”

“常数？你以为常数就不分大小了？”老勇者拿烟斗敲我脑袋，“集合查找确实是O(1)平均时间，但哈希运算、冲突处理、指针跳转，哪样不要CPU周期？你明明可以线性扫描一次边分类边统计，非要在那儿傻傻遍历26个字母，CPU的分支预测器都被你玩哭了，多出来的52简直是强迫症患者的噩梦。”

我顿时哑口无言，小声嘀咕：“那……我还有优化空间吗？”

老勇者哼了一声：“当然有。把你的世界观从‘能跑就行’升级到‘不给CPU添乱’，先试试分而治之。”

#### 第二战：双集合分类，扣点内存换条理

老勇者扔过来一个锦囊：“试试这个，分开关押，最后取交集。”

我展开一看，思路豁然开朗：遍历单词的时候，小写字母放左边口袋，大写字母转成小写后放右边口袋，最后取个交集，看谁两个口袋都有。

魔法阵重新绘制：

```python
lower = set()
upper = set()
for c in word:
    if c.islower():
        lower.add(c)
    else:
        upper.add(c.lower())  # 大写的间谍，先变成小写再装进口袋
return len(lower & upper)  # 两个口袋都有？那就是间谍！
```

这次我只用了一趟遍历，最后集合交集是C底层实现的，比之前一个个查快多了。

老勇者看着代码点点头：“嗯，有点意思了。空间上你整了两个集合，虽然还是O(n)，但逻辑清晰，代码可读性强。”

我兴奋得跳起来：“那我算不算入门了？”

老勇者吐出一个完美的烟圈：“入门？你还没摸到位运算的门槛呢。字母总共就 26 个，有没有办法用两个整数就把所有字母的存在性存下来？哪怕现在这 word长度只有50，位图操作依然是基本功。今天你偷懒用集合，明天遇到百万级流数据，内存和GC就能把你送走。”

我苦着脸：“位运算？就是那些>>、&、|？那不是大佬装逼专属吗？”

老勇者眼中精光一闪：“小子，看好了——我教你什么叫空间O(1)、寄存器级别的降维打击。”

#### 第三战：位运算大神的降维打击

老勇者在地上画了个大大的 0 和 1：“先让我们放大镜头，看看字符在计算机眼里的真实模样：”

| 字符 | ASCII码 | 二进制    |
| ---- | ------- | --------- |
| 'A'  | 65      | 0100 0001 |
| 'Z'  | 90      | 0101 1010 |
| 'a'  | 97      | 0110 0001 |
| 'z'  | 122     | 0111 1010 |

”发现彩蛋了吗？“ 他敲敲烟斗，“**秘密1**：大写字母的第6位（从右数，索引5）是**0**，小写字母的第6位是**1**。**秘密2**：同一个字母的大小写，**低5位完全相同**！（比如'A'和'a'的低5位都是`00001`）！”

我愣了三秒：“这……这难道是……”

“这就是字母的身份证！——当年设计ASCII码的那帮人早把位运算的接口给你留好了！”

他随手捻起一段咒语，正是传说中那让人头皮发麻的位运算代码：

```python
mask = [0, 0]  # mask[0] 存大写位图，mask[1] 存小写位图
for c in map(ord, word):
    mask[c >> 5 & 1] |= 1 << (c & 31)
return (mask[0] & mask[1]).bit_count()
```

我盯着这四行代码，大脑一片空白：“这……这是把代码压缩成.zip了吗？！”

老勇者哈哈一笑：“别慌，我一步步拆给你看。”

1. **准备动作：**

```python
mask = [0, 0]
```

“两个空整数，好比两张空白打卡表。mask[0]存大写字母的位图，mask[1]存小写字母的位图。现在 26 个位置全部是 0，将来哪个字母出现，就在对应的位置按个手印。”

2. **进入核心循环：**

```python
for c in map(ord, word):
```

“先用 `map(ord, word)` 把所有字符一键转成 ASCII 码。这是惰性求值，不会一次性生成列表，内存友好。省得每次都`ord()`拖慢速度——预处理优化，懂？”

我似懂非懂点头。

3. **决胜的魔法：**

```python
mask[c >> 5 & 1] |= 1 << (c & 31)
```

老勇者用树枝在地上划拉：“这一行干了三件事：① 区分大小写，② 提取字母编号，③ 按手印。全是寄存器操作，函数调用的开销连影子都没有。”

**第一件事：区分大小写 —— `c >> 5 & 1`**

“‘A’ 的 ASCII 是 65，二进制 `0100 0001`，右移 5 位变成 `0000 0010`，再和 1 做位与，结果是 0。所以选大写阵营`mask[0]`。”

“‘a’ 的 ASCII 是 97，二进制 `0110 0001`，右移 5 位变成 `0000 0011`，再与 1，结果是 1。选 小写阵营`mask[1]`。”

我恍然大悟：“原来 `>>5` 就是大小写鉴定器！大写得0，小写得1，精准分流！"

**第二件事：提取字母编号 —— `c & 31`**

“31 的二进制是 `0001 1111`，低 5 位全是 1。拿它跟任何 ASCII 码 按位与，相当于对32取模，把高位的 ‘大小写信息’ 掩掉，只留下低 5 位——也就是字母在字母表中的编号。比如 ‘A’ 是 65，编号`65 & 31 = 1`；‘a’ 是 97，编号`97 & 31` 也是 1。“

我啧啧称赞：”同一个字母的大小写形式，c & 31的结果一模一样！所以能保证后面'A'和'a'都会在各自阵营的位图同一位置按手印！”

**第三件事：按手印 —— `|= 1 << 编号`**

“左移`1 << 编号` 会得到一个只有在`编号`位是 1 的掩码，再用**位或赋值`|=` 把 mask 里对应的bit永久置1**。“

”这比集合的`add`操作轻了不止一个数量级——`add`要算哈希、处理冲突、可能扩容，而**位或赋值只是一条CPU指令**，直接在整数的某个bit上写1。”

我听得热血沸腾，老勇者干脆拉我坐下：“来，咱们跑一遍单词‘abBC’ 。”

举个栗子：初始：`mask[0] = 0b00000000`，`mask[1] = 0b00000000`

-  ‘a’（97）: `c>>5 & 1 = 1`小写，`c & 31 = 1`，`1<<1 = 2`，`mask[1] |= 2` → `0b0000 0010`
-  ‘b’（98）: 小写，编号 2，`1<<2=4`，`mask[1] |= 4` → `0b0000 0110`
-  ‘B’（66）: `c>>5 & 1 = 0`大写，编号 2，`1<<2=4`，`mask[0] |= 4` → `0b0000 0100`
-  ‘C’（67）: 大写，编号 3，`1<<3=8`，`mask[0] |= 8` → `0b0000 1100`

循环结束：`mask[0] = 0b0000 1100`（大写出现了 B、C），`mask[1] = 0b0000 0110`（小写出现了 a、b）。

最后一步：

```python
return (mask[0] & mask[1]).bit_count()
```

“`mask[0] & mask[1]` 按位与，得到 `0b0000 0100`——只有第 2 位同时为 1。`.bit_count()` 数出 1 的个数：1 个 ——只有字母b是双面间谍。”

“整个流程下来，空间只用了两个整数，64字节不到，时间复杂度O(n)且常数极小。没有函数调用开销，没有哈希计算，没有内存分配，CPU缓存友好得不得了。”

老勇者吐了个烟圈，“这就是位图法，嵌入式、数据库索引、布隆过滤器里都在用的套路。”

我瘫坐在地：“……合着我之前用集合简直是开着虚拟机跑原生代码，纯属给GC添乱？”

“倒也不是，”老勇者拍拍我，“这题n才50，三种方法都是毫秒级，但你得知道背后的代价——写业务代码的时候可读性优先，别上来就位运算糊脸，不然三个月后你自己都得用gdb单步调试。但在性能敏感或者数据量巨大的场景，能位运算就别用集合。一句话——你得知道你的代码在计算机肚子里是怎么跑的。”

#### 尾声：三大流派的终极感想

我拿出成绩单，村长盖了个大大的 “AC”。回望这次历险，我竟不知不觉掌握了三种姿态：

- **萌新粗暴流**：集合 + 遍历 26 字母。简单粗暴，小数据随便浪，交上去就能AC。但内存O(n)，时间复杂度O(n+52)，那多出来的52像是大O表示法被门夹过，常数因子属实不太体面。
- **工程分类流**：双集合取交集。逻辑清晰，只遍历一次，交集运算由C底层撑腰，干净利落，可读性好。就是空间还是O(n)，对内存稍微手松了点。
- **位运算降维打击流**：两个整数扛起26个布尔值，空间直接压到O(1)，CPU直呼内行，位操作一气呵成，寄存器里就把活干完了。缺点是可读性约等于加密，没注释就是精神攻击。

我问老勇者：“以后我该用哪种？”

他吐出一个大大的烟圈：“傻孩子，兵器没有绝对的好坏，看场合。日常搬砖，可读性优先；追求极致的赛场，位运算甩起来。但最重要的一点是——你要知道自己在干什么，而不是抄完代码就跑。”

我握紧手中那本开始翻毛边的魔法书，觉得自己离“算法型玩家”又近了一步。

而字母大陆的历险之旅，或许才刚刚开始。
