《字母大作战II：时空错乱的时间线危机》—— 当顺序成为致命武器

> 给你一个字符串 `word`。如果 `word` 中同时出现某个字母 `c` 的大写和小写，并且**每个小写 c 都出现在第一个大写 c 之前**，才叫特殊字母。返回特殊字母的数量。
>
> **彩蛋：字符串长度最多 2 × 10⁵**。

“二十万？”我手里的西瓜差点飞出去，“上次才50，这次直接上万马千军？这是DLC还是换引擎重制？”

村长拍拍我的肩，意味深长地说：“所以啊，别想着暴力莽过去。真正勇者，要经得起压测，不然服务器分分钟教你做人。”

我抹了抹嘴边的西瓜汁，深吸了一口气。

#### 第一战：按部就班，记录一切，最后算账

作为一个熟练掌握enumerate和字典查找的勇者，我坚信：**只要记录足够全，就没有查不出的真相**。

既然要求"所有小写都在第一个大写之前"，那我就分别记录：

- last_lower[c]：小写字母 c 最后一次出现的索引
- first_upper[C]：大写字母 C 第一次出现的索引

最后检查条件：last_lower[c] < first_upper[C]。如果成立，说明所有小写都在第一个大写之前，完美！

甩出两个字典，开干：

```python
last_lower = {}      # 记录每个小写字母最后出现的位置
first_upper = {}     # 记录每个大写字母第一次出现的位置

for i, c in enumerate(word):
    if c.islower():
        last_lower[c] = i          # 不断更新，保留最后一次
    else:
        if c not in first_upper:   # 只记录第一次
            first_upper[c] = i

cnt = 0
for c in "abcdefghijklmnopqrstuvwxyz":
    if c in last_lower and c.upper() in first_upper and last_lower[c] < first_upper[c.upper()]:
        cnt += 1
return cnt
```

“遍历一次单词再加26个字母检查，时间复杂度 O(n+26)……还行吧？”

刚嘀咕完，脑子里突然炸开老勇者的声音——就像某种 PTSD：

> “你明明可以线性扫描一次边分类边统计，非要在那儿傻傻遍历26个字母……”

我一拍大腿：“对哦！上次用两个口袋分别装小写和大写，边走边查，即时宣判。这次不就是多搞个口袋的事嘛！一旦发现‘大写后面又跟小写’的非法字母，就把它扔进黑名单口袋，最后整体剔除不就行了？”

#### 第二战：边跑边记，实时标记，最后过滤

我马上支棱起三个集合：`lower`（见过的小写）、`upper`（见过的大写，转小写存）、`invalid`（违规字母）。

规则很简单，实时审判：

- 遇到小写 `c`：加入 `lower`。如果 `c` 已经在 `upper` 里（说明之前见过它的大写），说明这个字母是“大写后面又来小写”——违规！丢进 `invalid`。
- 遇到大写 `C`：把它的小写形式加入 `upper`。

最后，符合条件的字母就是 `(lower ∩ upper) - invalid`，数一下长度完事。

```python
lower = set()      # 出现过的小写字母
upper = set()      # 出现过的大写字母（转小写存储）
invalid = set()    # 不合法的字母（大写后面又出现小写）

for c in word:
    if c.islower():  # 小写字母
        lower.add(c)
        if c in upper:  # 如果这个大写的后面有小写
            invalid.add(c)  # 标记为非法！
    else:  # 大写字母
        upper.add(c.lower())

# 从 lower 和 upper 的交集中去掉不合法的字母 invalid
return len((lower & upper) - invalid)
```

“漂亮！一趟扫描，实时揪出违规字母，最后集合运算一步到位。” 我恨不得把老勇者揪出来显摆，“ 老勇者，你看到高低也得点个赞吧？”

话音未落，一个烟圈慢悠悠飘过来。

老勇者果然出现了。他点点头，又摇摇头：“思路没毛病，但你还在用集合。”

“哈希表虽好，可咱昨天刚学过位运算，你咋就忘了？26个字母，用三个整数就能存下 `lower`、`upper` 和 `invalid`，寄存器级别的操作不香吗？”

#### 第三战：同样的逻辑，位运算实现，性能拉满

我猛拍脑门，恍然大悟："对对对！位运算！昨天两个整数扛起 26 个布尔值，今天再加一个整数当黑名单！我怎么把这神器给忘了！"

于是我立刻把集合换成了三个整数：

```python
lower = upper = invalid = 0
for c in map(ord, word):
    bit = 1 << (c & 31)  # 计算该字母对应的位掩码
    if c & 32:  # 小写字母（第6位是1）
        lower |= bit
        if upper & bit:  # 如果upper中也有这个字母
            invalid |= bit  # 标记为非法
    else:  # 大写字母
        upper |= bit
# 从 lower 和 upper 的交集中去掉不合法的字母 invalid
return (lower & upper & ~invalid).bit_count()
```

我得意地看着这三行核心逻辑，“空间三个整数，时间复杂度 O(n) 且常数极小。老勇者，这总该是天花板了吧？”

老勇者吸了口烟，缓缓吐出：“不错，位运算确实快。但你有没有注意到，每个字母只有四种可能的状态：‘只见过小写’、‘只见过大写’、‘已经配对成功’、‘已经违规’。”

“两个 bit 足够表示四种状态，你却用了三个整数，浪费了一个整数的感情。而且，你的答案在遍历结束之后才靠 `.bit_count()` 算出来——有没有想过，**在遍历过程中就实时维护答案**？”

我傻眼了：“这……还能升级？！”

#### 第四战：重新思考问题，用状态机实时维护答案

老勇者蹲下来，用树枝在地上画了一张状态转移图：

```
初始[状态0] --先见小写--> [状态1] --后见大写--> 合法[状态2]（计数+1）
  |                      |                    |
先见大写                又见小写               又见小写
  ↓                      ↓                    ↓
非法[状态-1]            不变[状态1]          非法[状态-1]（计数-1）
```

他一边敲烟斗一边解释：

- **状态0**：还没见过这个字母（初始）
- **状态1**：只见过小写，还没见过大写
- **状态2**：小写全在大写之前，配对成功（特殊字母）
- **状态-1**：违规（先大写后小写，或者配对成功后又在后面出现小写）

“用一个数组 `state[27]` 存每个字母的当前状态。遍历字符串时，每读一个字符就做一次状态转移，顺手维护一个 `ans` 计数器：该加一的时候加一，该扣掉的时候扣掉——**答案在遍历过程中就实时维护好了**——状态转移全是常数时间，连最后的 `bit_count` 都不需要。”

我瞪大眼睛看他写出代码：

```python
ans = 0
state = [0] * 27  # 26个字母的状态数组，索引1-26，索引0不用
for c in map(ord, word):
    x = c & 31  # 转成数字 1~26
    if c & 32:  # 小写字母
        if state[x] == 0:      
            state[x] = 1       # 0 -> 1
        elif state[x] == 2:    
            state[x] = -1      # 2 -> -1
            ans -= 1           # 撤销之前的计数
    else:  # 大写字母
        if state[x] == 0:      
            state[x] = -1      # 0 -> -1，先大写直接拉黑
        elif state[x] == 1:    
            state[x] = 2       # 1 -> 2，配对成功
            ans += 1           # 计数加1
return ans
```

我盯着代码，头皮发麻：“这……这比位运算还猛？连最后的 `bit_count` 都不用，直接实时维护答案？”

老勇者点点头：“ 位运算是把集合操作压缩到整数，本质上还是**事后计算**——跑完全程才出结果。而状态机是**在线维护**——每个字符过来，我只更新这个字母的状态，顺便把答案调好。如果输入流是无限的，位运算得等全部读完才能交差，而状态机却可以随时告诉你当前已经有多少个特殊字母。这就好比打游戏，你不需要等BOSS死了才知道自己打了多少输出，实时伤害统计一直挂在右上角。”

我看着 `state` 这个数组，有点疑惑：“可是 `state` 搞了 27 个 int 来维护状态，空间上怎么跟位运算比？位运算只用了三个整数，你这就翻了好几倍啊。”

老勇者听我问出这句话，满意地吐了个烟圈：“问到点子上了，小子。”

“`state[27]` 里存的是每个字母的当前状态。每个状态无非就四种取值：0、1、2 或者 -1，理论上两个 bit 就能塞下。但 Python 的 int 是个胖对象，底下还有引用计数和类型指针那一大家子，实际占的字节数比两个 bit 多得多——只不过，在分析算法空间复杂度的时候，我们看的是**逻辑存储量**，不是解释器的内存布局。”

他在地上画了两个圈：

“位运算那版，我用三个整数 `lower`、`upper`、`invalid` 存 26 个布尔信息，逻辑上一个整数扛起 26 个 bit 的位图，空间是 O(1)。状态机这版，我用一个长度为 27 的数组存每个字母的状态值，数组长度固定，不随输入规模 n 增长，所以**逻辑上也是 O(1)**。”

“一个用三个 int，一个用二十七个 int——从逻辑存储量来说，位运算流把每个字母压缩到三个 bit（三个位图各占一 bit），状态机流每个字母用了两个 bit（存四种状态）。数量级上都是‘常数个整数’，远没到需要掰扯谁更省的地步。真要抠这个常数的话……”他弹了弹烟灰，“位运算那版确实更省，三个整数 CPU 寄存器能直接抱怀里；状态机这版多出二十几个整数，还得通过数组索引去内存里捞，访存开销大了那么一丢丢。”

“但是，”他话锋一转，“状态机流的优势从来不在那几十个字节的内存上。它的杀招是**实时维护答案**——遍历过程中答案永远是最新正确的，不需要最后再调一次 `bit_count()` 去数。这叫用微不足道的空间常数差异，换了更灵活的时间策略。听懂了吗？”

我恍然大悟：“所以空间上它俩都是 O(1)，本质上是同一个重量级——都属于‘抠门流’。只不过位运算抠到极致，状态机则是抠得很聪明。”

老勇者点点头：“孺子可教。记住了，以后分析复杂度，先看量级再看常数。量级一样的时候，就别在几颗芝麻上纠结，去盯那些真正改变算法思想的地方。”

他站起身，拍了拍裤子上的灰：“最重要的，是看清问题的本质——这是一个**状态转移问题**。当你在做集合运算时，我已经在维护全局最优解了。状态机流之所以强大，不是因为它用了什么高深的数据结构，而是因为它重新定义了问题——不再是‘记录然后检查’，而是‘实时维护合法状态’。把离线批处理变成在线流式，才是真正的思想升维。”

#### 尾声：四大流派的终极顿悟

我拿着成绩单去找村长，老头眯着眼瞅了半天，大笔一挥又盖了个油光锃亮的“AC”。两场仗打下来，我竟然不知不觉攒出了一条清晰的成长线——

- **考古记录流**：字典存位置，最后统一算账。思路直白，时间常数里带着一股“暴力美学”的倔强。
- **三集合/三位运算流**：三个口袋边走边分类，用集合胜在逻辑清晰，用位运算则三个整数包打天下，空间压到O(1)。唯一的毛病是——答案得等遍历完了再用`bit_count()`数一遍。
- **状态机在线流**：一个数组存状态，遍历过程中实时加减答案。不用最后数bit，不用事后算账，每一步跑完答案都是最新正确的——就像游戏里挂着实时DPS统计。把“离线批处理”硬生生改成了“在线流式计算”，从“打完全局再复盘”进化到“边打边读秒”，直接来了个思想升维。

"原来，算法不仅仅是写出能跑的代码，更是思维的较量。3120教我学会了位图，用硬件的思维方式省内存；3121逼我画出了状态机，用数学的思维方式省时间。一个是在实现层面抠门，一个是在思想层面升维，两条路都能AC，但视野完全不一样。"

我握紧手中那本已经卷了边的魔法书，远处隐隐有新的卷轴光芒在闪动，不知道下一关又会掏出什么离谱条件。

字母大陆的试炼，果然每一次都让人热血沸腾。
