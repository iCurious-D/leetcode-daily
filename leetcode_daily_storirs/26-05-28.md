#### 第一战：暴力双循环，逐字符比后缀

我最先想到的，自然是照着题目描述硬来。

毕竟作为一个“能跑就行”型选手，我的座右铭是——“先把 AC 拿到手，优化的事情交给未来的自己”。

对于每个查询词，遍历一遍 `wordsContainer`，和每个候选词从后往前比，看公共后缀能走多长。同时记下最长匹配长度，再按题目那套“最短长度、最早出现”的奇葩优先级来更新答案。整个思路就跟手动翻字典一样朴实无华——查一个词就把整本字典从头翻到尾，查下一个再翻一遍：

```python
def stringIndices(self, wordsContainer: List[str], wordsQuery: List[str]) -> List[int]:
    ans = []
    for q in wordsQuery:
        best_idx = -1
        best_len = -1          # 最长公共后缀长度
        best_word_len = float('inf')  # 候选词本身的长度
        for idx, w in enumerate(wordsContainer):
            # 从后往前找公共后缀长度
            i, j = len(q)-1, len(w)-1
            common = 0
            while i >= 0 and j >= 0 and q[i] == w[j]:
                common += 1
                i -= 1
                j -= 1
            # 更新最佳答案
            if common > best_len:
                best_len = common
                best_idx = idx
                best_word_len = len(w)
            elif common == best_len:
                if len(w) < best_word_len or (len(w) == best_word_len and idx < best_idx):
                    best_idx = idx
                    best_word_len = len(w)
        ans.append(best_idx)
    return ans
```

样例一跑，好像没什么问题！我正打算提交，卷轴深处突然弹出了一个隐藏测试用例——`wordsContainer` 长度 10⁵，`wordsQuery` 长度 10⁵，每个词平均长度还不短。

这数据量一砸下来，我的代码立马就像一只误入高级副本的史莱姆，CPU 风扇开始咆哮，时间直接飙到天荒地老。

“啧啧，CPU 要是会说话，绝对要骂你是个没有感情的遍历机器。” 

老勇者不知从哪冒出来，吐着烟圈说：“你这一趟查询就要遍历整个容器，复杂度 O(n·m·L)，别说二十万，两千就能把你送走。你这哪是算法，你这是用生命在遍历。”

我苦着脸：“那怎么办？后缀这东西又不是前缀，字符串都是从前往后存的，总不能倒着建索引吧……”

老勇者眼中精光一闪：“倒着？你小子说到点子上了还没觉悟。”

“后缀匹配之所以难搞，是因为字符串从前往后存储，而你非要倒着比。如果把每个字符串反转一下，后缀就变成了前缀！而前缀匹配——正是 Trie 树的绝对主场。这就像你打魂系游戏，正面硬刚死活过不去，绕到背后一个背刺直接带走。”

#### 第二战：反转字符串 + Trie 树

**公共后缀 = 字符串反转后的公共前缀**：比如 `"abcdef"` 和 `"def"`，原生字符串的公共后缀是 `"def"`。反转后变成 `"fedcba"` 和 `"fed"`，公共前缀正好是 `"fed"`！

“妙啊！”我一拍大腿，“这样就能用前缀树——Trie 了！”

可我刚兴奋完又蔫了：Trie 树我懂，但题目里那套“最长、最短、最早”的奇葩优先级怎么塞进节点里？总不能每个节点里再开一个优先队列吧，那不成套娃了。

老勇者敲了敲烟斗：“你想复杂了。每个节点里只需要存两个值：**经过这个节点的、当前已知的最优候选词的（长度，下标）**。插入的时候，沿途**每个节点都试着更新这两个值——谁更短谁留下，一样短就看谁先来的**。”

“查询的时候，把查询词也反转，在 Trie 里能走多深走多深。走不动了就停，当前节点里预先存好的那个最佳下标，直接返回。不需要回溯，不需要比较，O(L) 走一趟就出结果。这就像每个路口都挂了块牌子：‘从这里往下走，最好的店在 3 号摊位’。你不用每家都逛，看牌子就完事！”

他随手在地上画了个 Trie 示意图：

```
假设容器词反转后是 "abcd"（原词 "dcba"）和 "ab"（原词 "ba"）。

插入 "abcd" (idx=0, len=4)：
路径节点 a->b->c->d 都会记录 (长度=4, 下标=0) 作为最佳。

插入 "ab" (idx=1, len=2)：
节点 a 和 b 会被更新为 (长度=2, 下标=1)，因为词更短。
节点 c 和 d 保持不变，因为 "ab" 走不到那里。

查询反转词 "ab"：
走到节点 b，返回存储的 (2, 1)，即对应原词 "ba" 的下标。。
```

我恍然大悟：“所以走到越深的节点，公共前缀自然越长，那个节点里存的最优候选词就是满足‘最长公共后缀’条件的词！插入时竞争上岗，查询时坐享其成——这不就是缓存的思想吗，提前把答案算好放着，随用随取！”

老勇者点点头：“你终于开始用脑了。”

**开始动手：Trie 节点的迷你竞技场**

我决定分两步走：

- **构建阶段**：将所有候选字符串反转后插入 Trie 树，沿途节点记录最优候选词。
- **查询阶段**：将查询字符串反转后在 Trie 树中查找，走到底直接返回预存下标。

先来个Trie 节点，长这样：

```python
class TrieNode:
    def __init__(self):
        self.children = {}           # 子节点字典：{字符: TrieNode}
        self.best_len = float('inf') # 该节点子树中最优候选词的长度
        self.best_idx = -1           # 该节点子树中最优候选词的下标
```

插入时，从根节点开始，每个经过的节点都擂鼓比武：当前这个词能不能把之前的冠军给KO掉？比赛规则就是题目那套——先看谁更短，一样短再看谁下标更小。而公共后缀长度这个维度——路径深度已经帮你筛过了——能走到同一个节点的候选词，公共后缀长度至少都等于当前深度，天然对齐。

再来个Trie树，长这样：

```python
class Trie:
    def __init__(self):
        self.root = TrieNode()       # 根节点（空字符）
        
    def insert(self, word: str, idx: int):
        node = self.root
        # 根节点也要更新，因为查询可能走到根就停了
        if len(word) < node.best_len:
            node.best_len = len(word)
            node.best_idx = idx
        for c in word:
            if c not in node.children:
                node.children[c] = TrieNode()
            node = node.children[c]
            # 沿途每个节点都是一场擂台赛
            if len(word) < node.best_len:
                node.best_len = len(word)
                node.best_idx = idx
                
     def query(self, word: str) -> int:
        node = self.root
        for c in word:
            if c in node.children:
                node = node.children[c]
            else:
                break   # 走不动了就停，当前节点存储的冠军就是答案
        return node.best_idx               

```

最后来个主函数，反转所有容器词插入，反转每个查询词查找，一气呵成

```python
def stringIndices(self, wordsContainer: List[str], wordsQuery: List[str]) -> List[int]:
    root = TrieNode()
    trie = Trie()
    for idx, w in enumerate(wordsContainer):
        rev_word = w[::-1]      # 反转插入，后缀变前缀
        trie.insert(rev_word, idx)
    res = []
    for q in wordsQuery:
        rev_q = q[::-1]         # 反转查询
        res.append(trie.query(rev_q))
    return res
```

我跑了一遍测试，毫秒级出结果。原本那个让暴力法陷入沼泽的 10^5 数据量，在 Trie 树面前就像热刀切黄油——顺滑得不像话。时间复杂度 O((N+Q)·L)，每个词只走一遍自己的长度，比暴力法的三重循环不知道高到哪里去了。

我正开心自己能从“能跑就行”选手升级为“跑得还行”选手的时候，老勇者又凑过来了。他瞟了一眼代码里的 `self.children = {}`，眉头一皱：“你用 Python 字典存子节点，虽然平均查找是 O(1)，但哈希计算、冲突处理、动态扩容，这些都是在堆里刨食。而且题目说了，只有小写字母——每个节点最多 26 个子节点，拿字典存简直是开哈希表去杀鸡。”

我瘫坐在地上：“......我就交个题而已，这么抠细节？”

“对于固定范围的小字符集，直接用**数组**代替字典。” 老勇者语重心长地说，“数组通过下标直接访问，连哈希计算都省了，CPU 缓存命中率还高。再加上 Python 的对象内存开销很大，每个 `TrieNode` 背后都拖着一个 `__dict__`，胖得跟没做优化的 Docker 镜像似的——这时候就该用 `__slots__` 给它物理瘦身，类实例不再维护属性字典，内存直接砍半。细节抠到位，才是真正的抠门大师。”

他吐出一个完美的烟圈，烟雾缭绕中写出了终极优化版。

#### 第三战：Trie 树完全体——哈希表到数组的瘦身革命

“看好了，这才是真正的 Trie 完全体——专为小写字母定制的极致抠门版。”老勇者用烟斗指着代码，一行行讲解起来，那架势就像在给新手村玩家展示满级神装。

```python
class Node:
    __slots__ = ('son', 'min_len', 'best_index')

    def __init__(self):
        self.son = [None] * 26        # 固定26个子节点，对应a-z
        self.min_len = float('inf')   # 子树中的最短字符串长度
        # best_index 不需要初始化，因为只在 min_len 更新时才赋值
```

我指着第一行就懵了：“`__slots__` 是什么玩意儿？看起来像是某种封印术，把对象属性锁死了。”

“没错，这就是给 Python 对象物理瘦身的禁术。” 老勇者深吸一口烟，“普通的 Python 对象背后都拖着一个 `__dict__` 字典来存属性，方便是方便，但每个对象都多占一两百字节内存。Trie 节点成千上万的时候，这开销比春节抢票的流量还吓人。`__slots__` 就是明确告诉解释器：‘这货只有这三个属性，别再给它配字典了’。内存直接腰斩，缓存命中率蹭蹭往上涨。”

他顿了顿，补充道：“还有，你注意看——`best_index` 我故意没在 `__init__` 里初始化。因为它只有在 `min_len` 被更新时才会赋值，之前根本访问不到。省一行是一行，抠门到骨头缝里，这才叫把每一滴性能都榨干净。”

还没等我吐槽 “你这已经不是写代码是在考古了”，他手里的烟斗又指向了 `self.son = [None] * 26`。

“这是第二个优化，也是最狠的一刀。之前你用字典 `{}` 存子节点，每次查找都要算哈希、处理冲突、还可能触发动态扩容，相当于每次找个字母都得去哈希表里翻箱倒柜。现在换成定长数组，`ord(c) - ord('a')` 直接算出索引——说白了就是内存偏移量的算术运算，指针一跳就到。这就像从‘查字典’升级到‘翻通讯录’：直接从第几页精准翻到第几个联系人，连目录都不用翻。”

“但一个节点就要固定开 26 个槽位，里面大部分可能是 `None`，这不浪费内存吗？”我忍不住提出质疑——毕竟我是被村长用“节省”二字反复 PUA 过的男人。

“这就叫空间换时间。”老勇者白了我一眼，“26 个指针而已，在现代 CPU 缓存行面前连塞牙缝都不够。况且，用字典的哈希表本身也有负载因子和桶数组的开销，实际内存未必比这省多少。在字符集已知的算法竞赛里，数组就是 Trie 的标配——你见过哪个打 ACM 的用 `dict` 存子节点？那相当于穿拖鞋去跑百米，能跑，但别人都穿钉鞋。”

我缩了缩脖子，继续往下看主逻辑：

```
class Solution:
    def stringIndices(self, wordsContainer: List[str], wordsQuery: List[str]) -> List[int]:
        ord_a = ord('a')
        root = Node()
        
        # 1. 构建 Trie 树
        for i, s in enumerate(wordsContainer):
            len_s = len(s)
            # 更新根节点（处理无匹配的情况）
            if len_s < root.min_len:
                root.min_len = len_s
                root.best_index = i
            
            cur = root
            # 直接逆序遍历，避免创建反转字符串
            for ch in reversed(s):
                c = ord(ch) - ord_a   # 字符转索引：'a'->0, 'b'->1...
                if cur.son[c] is None:
                    cur.son[c] = Node()
                cur = cur.son[c]
                
                # 维护当前节点的最优解
                # 因为 i 是递增的，所以只有当长度严格更短时才更新，保证了“长度相同取下标最小”
                if len_s < cur.min_len:
                    cur.min_len = len_s
                    cur.best_index = i		

```

“注意到没有，”老勇者敲了敲 `reversed(s)` 这一行，“这是第三个优化。之前你用的是 `w[::-1]`，看起来简洁，但切片操作会在内存里创建一整根全新的反转字符串，徒增 GC 压力。现在用 `reversed(s)` 迭代器，直接从后往前按需读原字符串，一个字符一个字符处理，零额外内存分配。在高频调用的场景下，这点省下来的时间和内存，可能就是压死骆驼的最后一根稻草——或者救活它的那口氧气。”

接着，他特意点出更新最优解的条件：“再来看这个 `if len_s < cur.min_len`。我只在长度**严格更短**的时候才更新 `best_index`。因为遍历容器时下标 `i` 是递增的，如果长度相同，当前这个词一定是下标更大的那个。按题目规则我们得保留更早出现的——也就是之前已经存好的那个，所以直接跳过就好，根本不用额外判断下标。这一个小细节，把‘长度相同取下标最小’的条件优雅地消化掉了，代码又省又干净，连个 `elif` 都不用写。”

最后是查询部分：

```
		# 2. 查询
        ans = []        
        for s in wordsQuery:
            cur = root
            for ch in reversed(s):
                c = ord(ch) - ord_a
                if cur.son[c] is None:
                    break
                cur = cur.son[c]
            # 退出循环时，cur 就是最长公共后缀对应的节点
            ans.append(cur.best_index)
        
        return ans
```

“查询没什么花活，跟之前逻辑一样——反转遍历，能走多深走多深，走不动了就停，当前节点里存的 `best_index` 直接就是答案。唯一的区别是子节点访问换成了数组索引，快了一个哈希函数的距离，顺便把 CPU 分支预测的坑也填平了。”

我盯着这段代码，感觉自己的灵魂都被优化了。这哪里是写代码，这简直是在雕刻艺术品！从 `__slots__` 内存瘦身，到定长数组 `self.son = [None] * 26` 的 O(1) 精准跳转，再到 `reversed` 迭代器的零拷贝遍历，每一处都在压榨最后一丝性能。原本庞大的 Trie 树现在像被施了空间折叠术，紧凑地蹲在内存里，随时准备以 O(L) 的速度解决每一个查询。

老勇者站起身，拍了拍裤子上的灰：“这个版本，时间复杂度还是 O((N+Q)·L)，但常数因子被压到了地板。空间上，每个节点固定 26 个指针加两个整数，又没了 `__dict__` 的拖累，内存占用大概只有字典版的一半。“

”如果你还想继续抠，可以把 `min_len` 用更小的类型存——但 Python 不给你这个机会，除非你上 C++ 或者 Rust，那就是另一个维度的神仙打架了。”

#### 尾声：三大流派的终极顿悟

我拿着通关成绩单找村长盖章，老头乐呵呵地给扣了个“AC”。走出村公所，我把战斗记录并排摊在草地上——暴力双循环、字典版 Trie、终极数组版 Trie。三张纸从皱巴巴到整整齐齐，像极了一个程序员的 commit history。

- **暴力倒比流：双重循环 + 逐字符比对。** 简单粗暴，逻辑直白。时间复杂度 O(N·Q·L)，数据量一上来就像用勺子挖英吉利海峡隧道，CPU 直接进入“冥想”态。适合面试热身或者本地跑个两位数样例，不适合任何有尊严的生产环境——除非你的用户清一色是树袋熊，对响应时间的感知以小时为单位。
- **反转 Trie 流：反转字符串 + 字典版 Trie。** 思维转换的典范，“后缀变前缀”直接把问题从“倒着比”的泥潭里捞出来，塞进前缀树的标准赛道。虽然用了哈希表存子节点，有常数开销，但可读性极佳，属于“面试官看了点头，同事 review 不会暗杀你”的工程优选。
- **终极数组 Trie 流：数组化 Trie + __slots__ 瘦身。** 这是为性能而生的完全体。定长数组干掉哈希计算，`__slots__` 封印 `__dict__` 的内存膨胀，`reversed` 迭代器省掉不必要的字符串拷贝。时间复杂度还是 O((N+Q)·L)，但常数因子被压到了地下室二层摩擦。

我盯着这三张纸，忽然觉得这远不只是一道题的解法递进。暴力法是“看见什么写什么”，反转 Trie 是“换个角度看问题”，终极优化是“把每一滴性能都榨出汁来”。三层境界，层层剥开指向同一个内核：**别让计算机替你干不该干的活。**

我把三张纸往怀里一揣，感觉自己不再只是一个会调库的脚本小子了。从位运算到状态机，从后缀反转到 Trie 瘦身，这几趟冒险教会我的不是某个具体 API 的用法，而是一种直觉——看到字符串就想“要不要倒过来试试”，碰到字符集固定就自动掏数组，创建大量对象时脑子弹出 `__slots__` 的优化选项，仿佛老勇者的烟斗随时会在耳边敲响。

远处，字母大陆的天际线又亮起一道新的传送门，光芒忽明忽暗，隐约能看到“后缀数组”之类的烫金大字在门楣上翻滚。那帮字母怪物不知道又在整什么离谱的新活。

“明日再战。”我朝传送门的方向挥了挥手，转身一屁股坐回村口的大槐树下。

西瓜还泡在冰水里，拎出来的时候瓜皮上滚着凉丝丝的水珠。一刀下去，咔嚓一声脆响，红瓤黑籽，甜得我差点忘了刚才差点被 O(n·m·L) 送走的事。
