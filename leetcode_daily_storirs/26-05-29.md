## 🎮《数位大冒险：最小元素挑战赛》

"小白，数字大陆的传送门打开了！” 今天一大早，我就收到了老村长发来的任务简报：“这片大陆上的数字们最近都膨胀了！它们不再是单纯的个位数，而是变成了多位数的怪物。你的任务是施展'数位分解术'，把每个数字打散成它的各个数位之和，然后找出这群家伙里最弱的那个！"

我揉着惺忪的睡眼扫了一遍任务简报，差点笑出声：“这不就是给史莱姆做体检吗？数数它们身上有几块粘液，挑最弱的那只揍。策划大概是怕我手生，特意在 Boss 战前塞了个教学关卡活动活动筋骨。”

任务卷轴上画着三波示例怪物，算是免费赠送的怪物图鉴：

第一波：`[10, 12, 13, 4]`

我抬手就是一记数位分解术——10 拆成 1+0=1，12 拆成 1+2=3，13 拆成 1+3=4，14 拆成 1+4=5。战后统计：`[1, 3, 4, 5]`，最小值 1。这波连热身都算不上，属于新手教程第一页。

第二波：`[1, 2, 3, 4]`

我盯着这几个数字愣了两秒——它们本来就是单个数位，怪物自己把血条贴在脑门上了，根本不需要动手分解。策划你是直接把测试用例的 placeholder 交上来了吧？最小值还是 1，毫无悬念。

第三波：`[999, 19, 199]`

终于来了几个看起来能打的。999 号恶魔 → 9+9+9=27，战力爆表，差点闪了我的血条。19 号狼人 → 1+9=10，中规中矩。199 号巫妖 → 1+9+9=19，看着唬人其实也就那样。战后统计：`[27, 10, 19]`，最小值 10，险胜。

#### 🔫 第一战：传统循环流——徒手拆数字

有图鉴做样本，我心里有底了。阿基米德曾说“给我一个支点，我能撬起地球”，而我想说：“给我一个取模运算符，我能把整个数字大陆拆成个位数。”

这种拆数字的活简直刻在肌肉记忆里：遍历每个怪物，掏出 `while num` 这把小刀——`num % 10` 精准切下末位数字，加到当前战斗力 `cur` 上，然后 `num //= 10` 把剩下的部分扔回去继续削。等怪物被削成光杆司令（变成 0），它的数位和就出炉了。最后拿 `min` 和全局血量 `ans` 比一比，更新最弱记录。

```python
ans = inf         # 初始化血量为无穷大
for num in nums:  # 遍历每个怪物
    cur = 0       
    while num:    
        cur += num % 10  
        num //= 10       
    ans = min(ans, cur)  # 记录目前遇到的最弱怪物
return ans  
```

‘辗转相除求数位和’——这不就是每个学 C 语言的孩子人生中写的第一个循环吗？时间复杂度 O(N·log₁₀M)，M 是数字的大小，在本题的数据量下绰绰有余。全程只做整数运算，不产生任何临时对象，寄存器友好得像是给 CPU 量身定制的按摩套餐。

但我盯着这七行代码，隐隐觉得它就像是用手摇发电机给手机充电——能用，但有点复古。for 里套 while，手动维护 `cur` 和 `ans`，写起来跟填表格似的。Python 好歹也号称“可执行的伪代码”，我写成这样，对得起那句“Life is short, I use Python”吗？

#### ⚡ 第二战：Pythonic 魔法流

我把魔法书往后翻了两页，决定换一套连招。既然 Python 内置了字符串和 `sum` 这一整套工具链，我干嘛还手动剥数字？这是脚本语言，不是嵌入式 C，该吃的语法糖就得吃——不然怎么好意思管自己叫 Pythonista。

新思路很直接：把每个数字用 `str()` 强转成字符串，等于把怪物大卸八块丢进一个字符数组；然后用 `map(int, ...)` 把每个字符挨个变回整数，散装成一个个独立的数位；`sum(...)` 大手一挥全给加起来。外面再包个生成器表达式惰性求值，最后 `min(...)` 直接锁定最小值。从头到尾一行代码，连临时变量都省了。

```python
return min(sum(map(int, str(num))) for num in nums)
```

“这简直就是‘一行流’玩家的福音！” 我盯着这行咒语，自己都忍不住感叹。类型转换、函数式编程、生成器表达式全揉在一起，像在玩《鬼泣》里一套空中连削带踩的华丽连招——看起来帅就完事，评分系统只算 Style Points，不管你是不是用力过猛。

这种写法才是真的 Pythonic——我只告诉计算机“要什么”，至于怎么拆、怎么加，解释器自己在 C 底层搞定。代码量从七行压缩到一行，可读性对写 Python 的人来说简直是母语级别。当然它也有代价：`str()` 会产生临时字符串对象，`map` 和生成器表达式也会多一丢丢函数调用开销，内存上比纯整数运算稍微大手大脚了一点。但本题的数据量 n 最多也就几百，每个数字撑死六七位，这点开销连给 GC 挠痒痒都嫌不够。要是换成百万级长度或者每个数字上千位的场景，那就得掂量掂量了——不过那是另一个副本的事，今天先不操心。

#### 🎮 通关结算

我提交代码，AC 两个字瞬间弹出来，快得像西瓜籽滑进喉咙，屏幕上甚至还跳了个“运行时间击败 99%”的彩蛋。我盯着那个数字愣了一秒——不是因为我写得好，是因为这题实在太水了，全站提交的人全挤在同一个毫秒级别里。

回看这两种流派，一套是徒手拆解流，整型运算直接刚正面，`while` 循环一把小刀削到底，全程不产生任何临时对象，适合喜欢掌控每一个时钟周期的硬核玩家。另一套是一键分解流，`str` 转 `map` 转 `sum` 再包生成器，一套函数式连招行云流水，适合信奉“能一行写完绝不写两行”的摸鱼术士。
