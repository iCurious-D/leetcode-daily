#### 💥 第一战：暴力流菜鸟的悲壮初体验

我——小白，昨天在数字大陆捏完各种史莱姆以后信心爆棚，今天拿到任务卷轴扫了一眼：“这不就是维护一个有序列表，每次查询时算最大间隙吗？难不倒我！”

本着“能暴力就绝不浪费脑细胞”的原则，我二话不说抄起 Python 内置武器库就开干：

```python
obstacles = []
ans = []
for q in queries:
    if q[0] == 1:
        obstacles.append(q[1])
        obstacles.sort()  # 每次插入都排序，任性！
    else:
        x, sz = q[1], q[2]
        obs = [0] + [p for p in obstacles if p <= x] + [x]
        max_gap = 0
        for i in range(len(obs) - 1):
            max_gap = max(max_gap, obs[i + 1] - obs[i])
        ans.append(max_gap >= sz)
return ans
```

样例一跑，哎哟喂，炸了！极限测试用例——五万条操作——我的CPU 风扇开始咆哮，屏幕上弹出一个血红的 **Time Limit Exceeded**，刺眼得像《黑暗之魂》里的 **“YOU DIED”**，鲜红刺眼，嘲讽拉满。

我瘫坐在地上，嘴里嘟囔：“逻辑没错吧？为什么超时？是不是评测机有问题？还是我没充会员？”

“用最勤快的遍历，挨最毒的打。”一个熟悉的声音从头顶传来。

老勇者叼着烟斗，从树后慢悠悠地飘出，“每次查询都要过滤、排序、再遍历，复杂度 O(Q² log Q)。五万乘五万，别说评测机了，你把这条代码扔到天河一号上去跑，它都得咳嗽两声。你这哪是算法，你这是给 CPU 上刑。”

#### 🌳 第二战：老勇者的线段树降维打击

老勇者用烟斗在地上画了一条数轴，标了几个点：“别在地上装死了。问题的本质很简单——障碍物把数轴切成了一段段空白区间。查询 [0, x] 就是问你：从起点到坐标 x 这段路上，最大的连续空白够不够塞进一个长度为 sz 的物块。”

“最大空白只可能来自两个地方：要么是完全落在最后一个不超过 x 的障碍物 `pre` 之前的某段完整间隔，要么就是 `pre` 到 `x` 这段最后的小尾巴。所以，我们只需要维护每个障碍物到它前一个障碍物的距离，然后找一个能支持**单点更新 + 前缀最大值查询**的数据结构——”

他磕了磕烟斗，火光中浮现出一棵倒挂的树：“线段树。”

“我们需要两个组件：一个能动态管理障碍物的有序集合、负责快速查找前驱后继的结构——Python里有现成魔法道具`SortedList` ；另一个就是能维护每个障碍物位置上的间隙值，支持单点修改和前缀最大值查询的数据结构——线段树。”

“插入障碍物 x 时，用 `SortedList` 找到它的前驱障碍物 `pre` 和后继障碍物 `nxt`，原本 `nxt - pre` 的完整间隙被切成了 `x - pre` 和 `nxt - x` 两段，在线段树上更新这两个位置的值。查询时，取 `max(前缀最大间隙, x - pre)`，一步到位。两个组件各司其职，全程 O(log n)，行云流水。”

一道魔法阵在烟雾中展开：

```python
def getResults(queries):
    mx = max(q[1] for q in queries) + 1
    seg = [0] * (2 << mx.bit_length())  # 满二叉树压位

    def update(o, l, r, i, val):
        if l == r:
            seg[o] = val
            return
        m = (l + r) // 2
        if i <= m:
            update(o * 2, l, m, i, val)
        else:
            update(o * 2 + 1, m + 1, r, i, val)
        seg[o] = max(seg[o * 2], seg[o * 2 + 1])

    def query(o, l, r, R):
        if r <= R:
            return seg[o]
        m = (l + r) // 2
        if R <= m:
            return query(o * 2, l, m, R)
        return max(seg[o * 2], query(o * 2 + 1, m + 1, r, R))

    obstacles = SortedList([0, mx])
    ans = []
    for q in queries:
        x = q[1]
        i = obstacles.bisect_left(x)
        pre = obstacles[i - 1]
        if q[0] == 1:
            nxt = obstacles[i]
            obstacles.add(x)
            update(1, 0, mx, x, x - pre)
            update(1, 0, mx, nxt, nxt - x)
        else:
            max_gap = max(query(1, 0, mx, pre), x - pre)
            ans.append(max_gap >= q[2])
    return ans
```

“线段树的 `update` 就像在一棵倒挂的二叉树上玩二分查找，每次把区间对半劈开，找到叶子节点完成赋值，然后递归返回的路上自底向上合并最大值——父节点永远是左右子节点的最大值，就像一个尽职的班长随时知道班里的最高分。”

“`query` 查询前缀最大值时，如果当前区间完全被 R 覆盖，直接返回现成答案，绝不逐片翻找。这种‘能整块拿就整块拿’的剪枝策略，正是线段树快到飞起的秘诀。”

果然，五万条查询丢进去，线段树连个嗝都不打，AC 弹出得干脆利落，像极了《只狼》里完美格挡后的忍杀特写——快、准、狠。

我一脸崇拜地准备收工，老勇者却一把拽住我：“别急着走，这个线段树虽然能 AC，但还是太‘重’了——又是递归又是维护满二叉树，常数不够极致。看好了，我再教你一种更骚的操作：树状数组 + 离线倒序。”

#### 🔥 第三战：树状数组 + 离线倒序——时间倒流的邪道打法

“离线？倒着处理？”我一脸懵，“这又不是拍《信条》，查询还能倒着做？”

“这就是思路升维。” 老勇者用烟斗在地上画出时间轴，“树状数组常数更小、内存更省，但它天生只擅长‘增量更新’——只能把值往上加，不能往下减。而这道题的障碍物偏偏是动态添加的，正向处理时想用树状数组，总感觉被绑住了手脚。所以，我们换个角度——把时间线翻过来。”

他深吸一口烟，写下新的流程：

1. 先一口气读完所有查询，把类型 1 的 x 全部提前塞进 `SortedList`，这就是‘最终状态’——所有障碍物都就位之后的完整景象。
2. 正向遍历一次这个有序列表，计算每个障碍物和它前一个障碍物的间隔，全部写进树状数组。这一步相当于在时间线的终点先拍了一张快照。
3. 然后**从最后一条查询开始，倒着往前处理**：
   - 遇到类型 2（查询）：和线段树版本一样，找到 `pre`，取 `max(前缀最大间隙, x - pre)`，一步到位，结果扔进答案数组。
   - 遇到类型 1（原本‘插入障碍物’的操作，在倒序的世界里摇身一变成了‘删除障碍物’）：从 `SortedList` 里移除 x，此时 x 的前驱 `pre` 和后继 `nxt` 会重新相邻。更新树状数组：把 `nxt` 位置的距离更新为 `nxt - pre`（因为 x 被删了，间隔合并了）。因为障碍物只删不增、间隔只会合并不会拆分，树状数组只需要支持单点更新成更大的值，这恰好是它最擅长的事，完全没有‘减小值’的烦恼。

终极咒语在烟雾中浮现：

```python
def getResults(queries):
    mx = 50000
    st = SortedList([0, mx])
    for q in queries:
        if q[0] == 1:
            st.add(q[1])

    bt = [0] * (mx + 1)

    def update(x, v):
        while x < len(bt):
            if v > bt[x]:
                bt[x] = v
            x += x & -x

    def query(x):
        res = 0
        while x > 0:
            if bt[x] > res:
                res = bt[x]
            x -= x & -x
        return res

    pre = 0
    for x in st:
        if x == 0:
            continue
        update(x, x - pre)
        pre = x

    ans = []
    for q in reversed(queries):
        if q[0] == 2:
            x, sz = q[1], q[2]
            idx = st.bisect_left(x)
            if idx < len(st) and st[idx] == x:
                pre_val = x
            else:
                pre_val = st[idx - 1]
            max_space = query(pre_val)
            max_space = max(max_space, x - pre_val)
            ans.append(max_space >= sz)
        else:
            x = q[1]
            idx = st.bisect_left(x)
            pre_val = st[idx - 1]
            nxt = st[idx + 1]
            update(nxt, nxt - pre_val)
            st.discard(x)

    return ans[::-1]
```

“树状数组的灵魂是 `x & -x`，提取二进制最右的那位 1——update 时用 `x += x & -x` 不断往父节点跳跃，把沿途节点的最大值更新；query 时用 `x -= x & -x` 一步步往左下方的区间跳，收集前缀最大值。”

”原本用线段树还需要来回递归的活儿，被树状数组用一套极其紧凑的 lowbit 跳跃全部搞定，全程无递归，全是迭代。整条时间线从后往前跑完，把攒在 ans 里的答案翻转一下，最终结果就此出炉——空间只开了一个长度为 mx+1 的数组，时间上连函数调用的栈帧开销都省了。

我瘫坐在地上，喃喃道：“先是暴力被教做人，然后线段树教我做人，现在树状数组又教线段树做人……算法的尽头不是更牛的数据结构，而是换个角度看问题啊。”

老勇者把烟斗往嘴里一叼，拍了拍裤子上的灰：“记住了，思维的高度，决定了你代码的优雅程度。以后碰到动态维护区间最大值的题，多问问自己——能不能离线？能不能倒着来？把插入变删除，把动态变静态，用时间换空间，用智商换常数。”

#### 🎮 通关结算

终于拿到通关“AC"以后的我决定复盘一下这次的战斗：

- **暴力扫描流**： O(Q² log Q)，五万数据直接升天。属于“我写着爽，CPU 跑到惨”。
- **线段树 + SortedList 流**：O(Q log M)，在线处理，动态维护间隔，工程上的优秀解。
- **树状数组 + 离线倒序流**：O(Q log M) ，但常数吊打线段树。插入变删除，时间线倒流——就像《信条》里子弹从墙上飞回枪口，反直觉，但优雅至极。

同一道题，从暴力到树状数组，表面上是数据结构的升级，骨子里是**问题建模方式**的进化。暴力法是“照章办事”，线段树是“用对的工具做对的事”，而倒序树状数组是“把问题翻个面，看见更简单的解法”。

我把魔法书往怀里一揣，数轴塔防已通关。下一关，策划又会掏出什么反直觉的骚操作？
