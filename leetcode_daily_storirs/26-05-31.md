## 🎮 行星吞噬者：从贪心到位运算的星际大胃王争霸赛

#### 🎯 任务简报

**副本名称**：星际大胃王 2126 - 摧毁小行星
**难度等级**：中等（贪心一把梭就能过，但想秀操作得掏出位运算）
**任务描述**：给你一个初始质量 `mass` 的行星，和一串小行星数组 `asteroids`。你可以按任意顺序安排撞击顺序——行星质量大于等于小行星时，小行星被摧毁，行星吸收它的质量；否则行星碎掉，游戏结束。问你能不能活着吃完所有小行星。

**任务奖励**：AC 一枚，贪心算法信仰值+1，位运算熟练度+2

------

#### 💥 第一战：直觉贪心流

昨天在数轴塔防里被线段树和树状数组轮流教做人的我，今天拿到任务简报，差点哭出来：“挑最小的先吃，吃完变大了再去啃大的，这不就是星际版贪吃蛇吗？终于来了一道不用建树的题了！”

贪心一把梭，证明交给数学老师，此时不上排序神器，更待何时！

```python
def asteroidsDestroyed(self, mass: int, asteroids: List[int]) -> bool:
	asteroids.sort()
	for a in asteroids:
		if mass < a:
			return False
		mass += a
	return True
```

数据范围 `10^5`，排序 O(n log n) 稳稳过关。但 AC 弹出来那一刻，我总觉得缺了点什么——这种“明明过了却感觉漏了什么”——果不其然，熟悉的声音从背后响起：

“ 排序确实是正确解法。Python 的 Timsort 是归并和插入的混合体，最坏也是 O(n log n)，稳定排序里的天花板，十万数据对它来说跟热身差不多。但是——” 

老勇者话锋一转，“质量范围才 10^5，你排的是序，浪费的是一个量级优化。计数排序听说过没？O(n) 就能搞定。”

我挠头：“计数排序？开一个十万长度的数组，统计每个质量出现了几次，然后按顺序遍历？可以是可以，但像个憨憨——我又不关心每个具体的质量值，只想知道当前最小的能不能吃、吃完能涨多少。”

“说得好。”老勇者点点头，“你真正需要的不是‘每个小行星排好队’，而是‘按量级分批处理’。看好了——用二进制位数当桶，直接把 O(n log n) 压成 O(n)。”

------

#### 🌳 第二战：按二进制位数分组

老勇者在地上画了几个圈：“你有没有注意到一个很有意思的性质？当你按质量从小到大吃小行星时，行星质量是单调递增的。一旦你的质量达到某个阈值，就能一口吃掉某个量级内的所有小行星。这个‘量级’，恰好可以用二进制位数来划分。”

他在圈里写着数字：“看好了，所有质量在 [1, 1] 的小行星，二进制只有 1 位；[2, 3] 是 2 位；[4, 7] 是 3 位；[8, 15] 是 4 位……每一组的下限恰好是 2 的幂，上限是 2 的下一个幂减一。这意味着我们可以按每个小行星的二进制位数分桶。Python 里 `bit_length()` 能直接告诉你这个数字属于哪个桶，相当于编译器免费帮你算好了桶编号。”

“这就好比给小行星们分宿舍——一共有 `max_width` 层，每层住着二进制位数相同的小行星。每层楼你只需要知道两件事：这一层最轻的住户有多重，以及整层楼吞下去总共能涨多少质量。因为一旦你的质量够到了这层的门槛（也就是最轻的那颗），整层楼就是你的自助餐了，先吃哪个后吃哪个根本无所谓。”

```python
def asteroidsDestroyed(self, mass: int, asteroids: List[int]) -> bool:    
    max_width = max(asteroids).bit_length()  # 确定分组桶数
    mn = [inf] * max_width   # 每桶中小行星的最小质量
    sum_ = [0] * max_width   # 每桶中所有小行星的总质量
    
    # 按二进制位数分桶
    for x in asteroids:
        i = x.bit_length() - 1  # 桶索引
        mn[i] = min(mn[i], x)
        sum_[i] += x
    
    for m, s in zip(mn, sum_):
        if m == inf:
            continue
        if mass < m:  # 无法摧毁这组的任意小行星
            return False
        mass += s     # 获得这组小行星的质量
    return True
```

“一次遍历分桶，再一次遍历按桶编号从小到大检查——门槛够不着就翻车，够了就整层吸收。虽然没有显式排序，但按二进制位数分组天然保证了‘先处理小的，再处理大的’的顺序。时间复杂度 O(n)，比排序少了一个 log n 的因子，常数也更漂亮。”

我盯着这段代码，正准备喊“妙啊”，忽然皱起了眉头：“等等，这里用了两个长度为 `max_width` 的数组。现在整个小行星带里最大那颗是 10^5，也就 17 个二进制位，数组才 17 个元素，倒不算浪费。但如果数据很稀疏——比如有一颗 2^30 级别的巨无霸混在一堆小不点里——那这个数组不是大部分都空着吗？就像修了一栋摩天大楼，结果只住了三层人。”

“问到点子上了。”老勇者眼中闪过一丝赞许，“数组虽好，在稀疏场景下确实会浪费空间和遍历时间。如果你想写一套在任何数据范围下都紧凑且优雅的方案——用字典加位掩码。”

#### 🔥 第三战：稀疏分组：字典 + 位掩码 

“数组版本的缺点刚才你也说了——如果数据稀疏，定长数组就像修了一栋摩天大楼结果只住了三层人，空间和时间都在为空气打工。所以换个思路：只给有人的楼层开房间，再用一个整数的二进制位当楼层指示灯。这就是字典加位掩码的组合拳。”

”用 `defaultdict` 代替定长数组，空间随实际出现的组数动态增长。同时用一个整数的二进制位来标记哪些组出现过——这就是位掩码的老本行。具体可以二步走：
第一步，分桶标记。遍历所有小行星，每个算 `bit_length` 找到楼层号，更新字典里对应楼层的 `mn` 和 `sum_`，同时把 `mask` 的对应位置 1——相当于那层楼的灯亮起，表示‘有人住’。
第二步，循环按序提取。用 `mask & -mask` 不断提取最低位的 1，这招就是树状数组里的 lowbit，能精准定位到当前编号最小的亮灯楼层，把它还原成楼层号。然后判断吸收，也就是取出这层的门槛和总质量，如果当前行星质量连门槛都够不着，直接返回 `false`；够了就整层吞下，然后用 `mask ^= lowbit` 把这层的灯熄灭。循环直到 mask 归零，全部通过就返回 `true`。”

“整段代码里，字典只存有值的楼层，位掩码保证遍历顺序严格按楼层编号从小到大。就像游戏里只渲染地图上有资源点的迷雾区域，空白地带连个 draw call 都不浪费。”

```python
def asteroidsDestroyed(self, mass: int, asteroids: List[int]) -> bool:
    mn = defaultdict(lambda: inf)  # 创建默认字典，访问不存在的键时返回无穷大
    sum_ = defaultdict(int)        # 创建默认字典，访问不存在的键时返回0
    mask = 0                       # 位掩码，组出现的登记表

    for x in asteroids:
        i = x.bit_length() - 1
        mn[i] = min(mn[i], x)
        sum_[i] += x
        mask |= 1 << i  # 将第i位设置为1，标记该组存在小行星

    while mask:
        lowbit = mask & -mask        # 提取mask中最右边的1（最低有效位）
        i = lowbit.bit_length() - 1  # 转换回组索引
        if mass < mn[i]:  # 无法摧毁这组的任意小行星
            return False
        mass += sum_[i]  # 获得这组小行星的质量
        mask ^= lowbit               # 移除mask最低位的1
        
    return True
```

“`mask & -mask`！”我差点从地上蹦起来，“这不就是昨天树状数组里那招 lowbit 吗！一模一样！”

老勇者瞅了我一眼，一副“学费没白交”的表情：“对，同门师兄弟，专门用来精准定位最低位的 1。`mask & -mask` 提取最低位的 1，`mask ^= lowbit` 用异或把这个位清零——比减法安全，比取反掩码再与简洁，一条 CPU 指令就能搞定的事，绝不写两行。位掩码就像一个智能指针，每次自动指向下一个非空组，保证按从小到大的顺序处理，完全没有浪费时间在空组上。“

“字典只存实际出现的组，位掩码保证遍历顺序严格按组编号从小到大。整段代码没有排序，没有定长数组，时空复杂度都只跟实际出现过的二进制位数有关——妥妥的 O(n) 时间加稀疏级别的空间。”

我佩服得五体投地：“我以为贪心排序就是最优解了，没想到按位分组能压成 O(n)，更没想到最后位掩码加字典连定长数组都省了。除了‘门槛’和‘收益’，行星中间的顺序细节全是冗余信息。”

说到这儿我忽然愣住，脑子里一道闪电劈过：“等等，这不就是数据库索引里 B+树的精髓吗？非叶子节点只存每个子块的最小键值和指针，不存具体数据，查的时候先定位到块，再在块内细搜。我们这里每层楼的最小质量就是那个‘最小键值’，整层的总质量就是‘指向那组数据的指针’——不需要遍历每一颗小行星，只需要知道每个量级的门槛和总量，贪心一把全收走！”

老勇者哈哈一笑：“能悟到这一层，孺子可教也。算法优化的本质其实也就是在不断追问——哪些信息是冗余的，哪些操作是可以合并的，哪些约束是可以绕过不看也能拿到最优解的。”

------

#### 🎮 通关结算

拿到 AC 的我，照例把战斗记录往草地上一摊：

- **贪心排序流**：先整理再动手，O(n log n)，直白稳妥，面试首选。

- **按位分组流**：抓住本质，用二进制位数当桶，只记最小值和总和，O(n) 时间，O(log M) 空间。
- **位掩码 + 字典流**：字典只存非空组，lowbit 当智能指针精准跳转，O(n) 时间，连定长数组都省了。

盯着这三个记录，我忽然乐了。从昨天数轴塔防的树状数组到今天的位掩码，`lowbit` 这招连着两天在我脸上刷存在感。昨天用它爬树，今天用它定位，同一个 `x & -x`，换个副本就是完全不同的用法。

学的时候以为是背招式，用多了才发现，练的其实是同一套内功。
